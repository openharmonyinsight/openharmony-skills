# OpenHarmony Agent Skill 实践指导书

## 0. 规范优先级与术语约定

### 0.1 规范用词

本指南中的规范词语含义如下：

- `必须`：不满足则不得发布，除非评审记录中明确批准例外
- `应当`：默认要求；若不满足，应在评审记录中说明理由
- `建议`：推荐做法；不作为单项否决依据
- `可`：按任务需要选择

### 0.2 术语约定

除非另有说明，本指南中的 `Agent` 统指各类 Coding Agent。

### 0.3 规范优先级

规范冲突时按以下顺序处理：

1. `Agent Skills` 开放规范
2. 本组织级编写规范
3. `skill-creator` 当前工作流与工具实现

其中：

- `Agent Skills` 开放规范约束 Skill 的兼容性交付要求
- 本指南约束组织内部质量标准
- `skill-creator` 用于提高创建、检查与验证效率，但不得覆盖前两者

若三者出现冲突，应当在评审记录中注明冲突点、处理依据与最终结论。

## 1. 目的与适用范围

本指南用于统一组织内部 Skill 的创建、重大修改、评审与发布标准，目标是让 Skill 在兼容平台上具备以下特征：

- 可触发：Agent 能准确判断何时应使用该 Skill
- 可执行：Agent 读完后能采取正确行动，而不是只得到泛泛建议
- 可复用：内容可跨项目、跨任务重复使用
- 可迁移：交付物尽量不依赖某一平台的专有机制
- 可维护：体量受控、结构清晰、便于迭代

本指南以 `Agent Skills` 开放规范为基线，并吸收 `skill-judge` 的评价思路，形成组织内部的统一质量标准。

本指南适用于拟纳入组织库的 Skill。一次性实验性 Skill、个人临时 Skill 不在强制范围内，除非明确进入组织库或被要求按组织标准维护。

## 2. 编写前判断

### 2.1 什么时候应当写成 Skill

满足以下一种或多种情况时，应当优先考虑写成 Skill：

- 需要将一套可复用的方法稳定传递给 Agent
- 同类任务会反复出现
- 任务成败依赖判断经验，而非单纯事实记忆
- 团队希望统一某类工作的做法和质量标准
- 存在组织特有上下文，无法依赖通用模型自行补全

### 2.2 什么时候不应当写成 Skill

出现以下情况时，不应当优先写成 Skill：

- 仅适用于单一项目或单次交付的一次性说明
- 更适合通过代码校验、脚本、规则引擎、模板或产品能力解决
- 只是把已有文档原样搬运进来
- 主体内容是基础知识普及，没有显著知识增量

在决定写 Skill 前，应当先判断该问题是否更适合由代码校验、模板、规则引擎、产品能力或普通文档解决，避免把所有问题都 Skill 化。

### 2.3 设计原则

#### 2.3.1 知识增量优先

Skill 的价值应主要来自专家知识增量，而非重复模型通常已知的通用常识。

高价值内容通常包括：

- 组织内部约定、标准、术语和流程
- 领域特有的判断框架、决策树和取舍原则
- 高价值反模式、常见误判和边界条件
- 经过验证的可复用做法

以下内容不应成为 Skill 主体：

- 教科书式概念介绍
- 通用工具基础用法教程
- 大模型本就熟悉的常识性流程
- 没有判断价值的操作清单

提示语：

`好 Skill = 专家知识增量 - 模型已知常识`

#### 2.3.2 触发设计优先于知识堆砌

Skill 首先要解决“什么时候应被调用”，其次才是“调用后做什么”。若 `description` 和正文都无法清楚表达适用场景，则该 Skill 不合格。

#### 2.3.3 渐进披露

正文应当仅保留每次加载都需要的核心指令。较重的参考材料、案例、脚本和模板，应当放在独立文件中，并在正文中明确写出“何时读取哪个文件”。

#### 2.3.4 自由度与任务脆弱性匹配

- 创造性、高变动任务：应当给原则、判断框架和推荐路径
- 半结构化任务：应当给步骤、分支条件和关键约束
- 易错、脆弱、后果高的任务：应当给更明确的执行顺序、参数和校验要求

#### 2.3.5 平台中立

平台中立要求主要约束 Skill 交付物本身，尤其是 `SKILL.md` 正文内容；不禁止组织在内部生产流程中指定统一的辅助工具或校验流程。

除非 Skill 的目标本身就是某个平台或某类工具链，否则 `SKILL.md` 正文不应绑定特定 Agent 平台的专有能力、专有模式、专有目录约定或仅在该平台成立的上下文假设。

## 3. 编写要求

### 3.1 编写流程

组织内所有新建或重大修改的 Skill，必须遵循以下流程：

1. 明确目标任务、典型输入、期望输出、触发语、反例和不适用边界
2. 使用 `skill-creator` 创建或更新 Skill 骨架
3. 按 `Agent Skills` 开放规范编写 `SKILL.md`
4. 按本指南完成自检与修订
5. 使用 `skill-creator` 完成检查与快速验证
6. 通过本指南第 4 节的准入检查
7. 经本指南第 5 节评分达到发布门槛后纳入组织库

### 3.2 关于 `skill-creator` 的组织要求

`skill-creator` 是组织内部 Skill 生产流程的一部分，要求如下：

- 新 Skill 必须通过 `skill-creator` 初始化
- 修改现有 Skill 时，应当参考 `skill-creator` 的当前工作流，而不是沿用过时习惯
- 提交评审前，必须使用 `skill-creator` 执行一次检查或快速验证
- 评估 Skill 时，应当同时检查其是否符合 `skill-creator` 所强调的核心原则：简洁、适当自由度、渐进披露、真实使用验证

若所在平台暂不支持 `skill-creator`，应当在评审记录中说明，并由评审人按本指南补做等价检查。

### 3.3 标准目录与文件结构

本指南不重复定义标准目录结构。目录、必需文件、可选资源目录与引用方式，一律以 `Agent Skills` 开放规范为准。

组织内仅补充以下要求：

- 不要为了“看起来完整”而引入不必要的目录层级
- 参考文件必须能从 `SKILL.md` 直接按条件定位，不得成为无人加载的“孤儿文档”

### 3.4 `SKILL.md` 的强制要求

#### 3.4.1 Frontmatter

`SKILL.md` 必须包含 `Agent Skills` 开放规范要求的最小元数据，至少保证：

- `name` 准确、简洁、可检索
- `description` 明确说明何时使用，不写成泛泛能力介绍

`description` 应当满足以下要求：

- 以适用条件为中心，而不是以功能宣传为中心
- 覆盖任务触发词、症状词、场景词和常见同义表达
- 不复述完整流程，不把正文摘要塞进描述
- 不出现平台专有术语，除非 Skill 本身就是面向该平台
- 足够具体以支持触发判断，但避免堆入完整流程或大段背景说明

不佳示例：

> 帮助用户完成数据分析任务，提高效率。

问题：

- 不说明何时触发
- 没有任务边界
- 没有症状词或场景词
- 像能力宣传，不像触发条件

改写方向：

> Use when the task requires choosing analysis method, validating assumptions, or interpreting ambiguous business metrics under organization-specific reporting rules.

#### 3.4.2 正文必须回答的问题

一个合格的 `SKILL.md` 至少应当让 Agent 在加载后回答清楚以下问题：

- 这个 Skill 解决什么问题
- 遇到什么信号时应触发它
- 首先该判断什么
- 推荐采取哪些步骤或策略
- 哪些做法绝对不要采用
- 出现分支、异常或信息不足时如何处理
- 什么时候要继续读取额外参考文件

#### 3.4.3 最小可用骨架

作者应当至少按以下骨架组织正文；具体标题可调整，但信息职责不得缺失。

```markdown
---
name: ...
description: ...
---

# Task and Boundaries
说明解决的问题、输入输出边界、不适用情形

# Trigger Signals
列出应触发的典型任务、症状词、同义表达

# Initial Checks
收到任务后先检查哪些条件，按什么顺序判断

# Execution Strategy
给出核心步骤、分支条件、关键约束

# Prohibited Practices
列出高风险反模式及原因

# Exceptions and Fallbacks
信息不足、条件不满足、执行失败时如何处理

# References
说明何时读取哪个文件
```

#### 3.4.4 内容编写规则

- 应当优先写判断依据，不要只写动作步骤
- 应当优先写高价值边界条件，不要堆积低价值背景知识
- 若存在多路径选择，必须给出选择条件，而不是并列罗列做法
- 反模式必须具体，并说明为什么错
- 示例应当短小，只服务于规则理解，不要写成教程

不佳示例：

> 先尝试方法 A，也可以试试方法 B，必要时再考虑方法 C。

问题：

- 没有说明选择条件
- 无法支持 Agent 做稳定判断
- 更像头脑风暴，不像可执行规范

改写方向：

> 默认使用方法 A；仅当输入数据不完整时改用方法 B；若任务目标要求可追溯审计，则必须使用方法 C。

### 3.5 内容设计要求

#### 3.5.1 写出“如何思考”

高质量 Skill 不只是告诉 Agent “做什么”，还应当告诉 Agent “先看什么、如何判断、何时切换路径、何时停止”。

正文应当优先包含：

- 决策顺序
- 取舍标准
- 典型误判
- 边界与止损条件

#### 3.5.2 控制篇幅与加载成本

- 正文只保留高频必读内容
- 重型资料应迁移到外部引用文件
- 长文档必须具备明确读取条件
- 若一段内容删除后不影响正确决策，应优先删除
- 若某段内容只在少数分支场景下使用，应迁移到引用文件，并在正文中写明读取条件

#### 3.5.3 保持内容与平台解耦

正文避免出现如下内容：

- 特定 Agent 平台的调用机制
- 特定工具名或模式名，除非任务本身依赖该工具
- 仅在某平台成立的记忆、上下文或权限假设

## 4. 准入检查

本节用于判断一个 Skill 是否具备进入正式评审的最低条件。准入检查是布尔判断，不用于区分质量高低。

以下条件必须全部满足，方可进入评分阶段：

- 该 Skill 解决的是复用问题，不是一次性问题
- 主体内容以知识增量为主，而非基础教学
- `description` 可用于触发，不是营销式描述
- 正文包含任务边界、触发信号、首要判断、执行策略、反模式和回退机制
- 没有不必要的平台专有依赖
- 目录结构遵循 `Agent Skills` 开放规范
- 参考文件都有明确读取条件
- 已通过 `skill-creator` 创建或完成等价检查

以下情况属于阻断发布项，命中任一项则不得发布：

- frontmatter 不可触发或明显误导
- 正文缺少可执行路径
- 无法识别不适用情形
- 存在明显平台绑定且未说明必要性
- 参考文件无读取条件，形成无效堆料
- 与 `Agent Skills` 开放规范冲突且无解释

## 5. 评分与发布

### 5.1 评分方法

组织内评审采用以下 8 个维度。评分用于判断质量水平，不替代第 4 节的准入检查。

建议使用四级评语：

- `优秀`：可直接纳入组织标准库
- `良好`：允许发布，但需记录改进项
- `待改进`：需修订后复审
- `不合格`：应重写或拆分

### 5.2 评分维度

#### D1. 知识增量

检查该 Skill 是否真正提供了专家级信息，而不是重复模型常识。

高分表现：

- 提供非显而易见的判断框架
- 包含真实场景中的约束、取舍和边界条件
- 能明显减少 Agent 在该类任务上的试错

低分表现：

- 主要内容可由通用模型直接给出
- 缺少组织特有约束
- 没有体现取舍、边界或误判修正

分级锚点：

- `优秀`：明显提供非通用判断框架，可直接提升该类任务成功率
- `良好`：包含一定知识增量，但边界条件不够充分
- `待改进`：有部分定制内容，但主体仍偏常识
- `不合格`：几乎是通用教程或空泛流程

#### D2. 心智模型与程序性知识

检查该 Skill 是否传递“如何思考”和“如何推进”的方法，而不仅是概念解释。

高分表现：

- 有先后顺序明确的判断路径
- 有任务分支条件
- 有针对领域问题的非通用工作法

低分表现：

- 只有名词解释，没有行动顺序
- 给了步骤，但看不出为什么这样做
- 出现分支场景时无法指导选择

分级锚点：

- `优秀`：兼具判断路径、分支条件和推进方法，Agent 可据此独立完成任务
- `良好`：存在主要路径，但异常和分支处理不够充分
- `待改进`：有一些步骤，但缺少稳定的思考框架
- `不合格`：基本停留在概念说明层

#### D3. 反模式质量

检查是否明确指出高风险错误做法，以及这些做法为什么会失败。

高分表现：

- 反模式具体，不空泛
- 带原因说明，而不是只有禁令
- 与真实失败模式强相关

低分表现：

- 反模式只有口号，没有场景
- 只写“不要这样做”，未说明后果
- 反模式与实际失败案例脱节

分级锚点：

- `优秀`：反模式具体、常见、可直接降低错误率
- `良好`：有实用反模式，但覆盖面有限
- `待改进`：反模式存在但过于抽象
- `不合格`：缺少反模式或基本无指导价值

#### D4. 规范符合度

检查是否符合 `Agent Skills` 规范，尤其是元数据、正文职责和引用方式。

高分表现：

- 元数据准确且可触发
- `SKILL.md` 结构清晰
- 标准结构与引用方式符合开放规范

低分表现：

- frontmatter 不完整或不利于触发
- 正文职责混乱
- 引用结构与 `Agent Skills` 开放规范明显不一致

分级锚点：

- `优秀`：兼容性和组织规范均符合，几乎无歧义
- `良好`：基本符合，存在少量可修正问题
- `待改进`：有多处偏差，需要集中修订
- `不合格`：核心结构或元数据不达标

#### D5. 渐进披露

检查内容是否分层得当，避免把全部知识塞进主文档。

高分表现：

- 主文档轻量
- 参考材料按条件读取
- 无“孤儿文件”或“无条件全量加载”倾向

低分表现：

- 主文档堆满低频信息
- 重型资料没有拆分
- 参考文件存在但没有读取条件

分级锚点：

- `优秀`：正文与参考材料分层清晰，加载成本低
- `良好`：整体分层合理，个别内容仍可拆分
- `待改进`：已有拆分，但读取条件不清或主文仍偏重
- `不合格`：基本无分层意识

#### D6. 自由度校准

检查指导的精细程度是否和任务脆弱性匹配。

高分表现：

- 创造性任务不过度锁死
- 高风险任务不过度放任
- 指令既不僵化，也不失焦

低分表现：

- 对高风险任务给出过于宽泛的建议
- 对开放任务给出僵硬路径
- 无法看出哪些规则是硬约束，哪些是建议

分级锚点：

- `优秀`：约束强度与任务风险高度匹配
- `良好`：整体合理，少量部分过宽或过紧
- `待改进`：存在明显自由度失配
- `不合格`：指导方式与任务特性明显冲突

#### D7. 模式合理性

检查该 Skill 的组织方式是否符合其任务类型。

高分表现：

- 结构与任务复杂度匹配
- 没有为简单问题设计过重结构
- 没有把复杂流程压缩成无法执行的口号

低分表现：

- 简单问题写得过重
- 复杂问题写得过轻
- 章节组织不支持实际使用

分级锚点：

- `优秀`：结构天然服务于任务执行
- `良好`：结构基本合理，局部还可优化
- `待改进`：结构与任务复杂度存在明显偏差
- `不合格`：结构失衡，显著影响使用

#### D8. 实用可用性

检查 Agent 拿到 Skill 后是否真的能用起来。

高分表现：

- 触发条件清楚
- 执行路径可落地
- 包含异常处理、回退或升级条件

低分表现：

- 看起来完整，但 Agent 仍无法开始执行
- 没有异常处理或回退路径
- 写法更像理念宣言，而不是工作指南

分级锚点：

- `优秀`：Agent 可直接按文档稳定执行
- `良好`：主路径可执行，异常场景还可加强
- `待改进`：局部可用，但需作者大量口头补充
- `不合格`：难以直接使用

### 5.3 发布门槛

组织内建议采用以下门槛：

- `D1 知识增量` 不得低于 `良好`
- `D4 规范符合度` 不得低于 `良好`
- `D8 实用可用性` 不得低于 `良好`
- 任一维度若存在明显缺陷，不得以平均结果掩盖

## 6. 模板与附件

本指南不在正文中展开模板细节。标准模板、评审表和示例应作为附件单独维护，避免规范正文与具体样板耦合。

当前建议至少维护以下附件：

- Skill 模板附件
- 评审评分表附件
- 优秀 Skill 示例附件
