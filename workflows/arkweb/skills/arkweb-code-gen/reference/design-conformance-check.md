# 设计遵从性检查方法

本文件定义设计遵从性门禁的比对维度、分类规则和处理流程。
由 Team Lead 在 Phase 2 执行时读取。

## Contents

- 比对维度（5 项）
- 偏离分类规则（3 级）
- 处理流程
- 偏离报告输出格式

## 比对维度

从设计文档中提取以下强制约束项，与分析报告中的实现方案逐一比对：

| 维度 | 比对方法 | 示例 |
|------|----------|------|
| 技术路径/架构方案 | 设计文档指定的实现方式 vs 分析报告的方案选择 | 设计要求 C++ DOM API vs 分析报告采用 JS 注入 |
| 错误码体系 | 设计文档定义的错误码 vs 分析报告使用的错误码 | 设计定义 -1~-10 专项码 vs 分析复用已有 102/131/132/133 |
| 实现范围（涉及的层级/仓库） | 设计文档指定的所有层级 vs 分析报告覆盖的层级 | 设计要求修改 ACE Engine vs 分析未涉及 ACE Engine |
| 接口定义/API 签名 | 设计文档的接口设计 vs 分析报告的接口方案 | — |
| 参数校验规则 | 设计文档的校验要求 vs 分析报告的校验实现 | — |

## 偏离分类规则

对每个偏离项，按以下标准分类：

### [偏离-需用户确认]（阻断流程）

- 技术路径与设计文档明确指定的方案不同（如 C++ DOM API 被替换为 JS 注入）
- 实现范围缩减（设计文档要求的层级/模块被遗漏）
- 错误码体系未按设计文档定义
- 设计文档中标注为"强制"/"必须"的约束未被遵守

### [偏离-可接受]（继续流程，但需记录）

- 文件命名微调（符合项目惯例）
- 补充了设计文档未提及但项目规范要求的特性（如 Feature Flag）
- 实现细节的合理调整（不影响架构和接口）

### [一致]

- 分析报告与设计文档完全一致

## 处理流程

### 存在 [偏离-需用户确认] 项

1. 停止流程，不进入 Phase 3
2. 向用户输出偏离报告（见下方格式）
3. 等待用户确认后决定是否继续

### 仅有 [偏离-可接受] 或全部 [一致]

1. 记录可接受的偏离项（在后续 Phase 7 中纳入差异记录）
2. 正常进入 Phase 3

## 偏离报告输出格式

```
┌──────────────────────────────────────────────────┐
│  Phase 2: 设计遵从性检查未通过                    │
├──────────────────────────────────────────────────┤
│  发现以下偏离需用户确认：                         │
│                                                  │
│  偏离 1: [偏离标题]                              │
│  设计文档要求：[设计文档中的方案]                  │
│  分析报告方案：[分析报告的方案]                    │
│                                                  │
│  偏离 2: ...                                     │
│                                                  │
│  请确认：                                        │
│  A) 接受偏离，按分析报告方案继续                  │
│  B) 拒绝偏离，退回 Phase 1 重新分析               │
│  C) 逐项决定（告知每项选择）                      │
└──────────────────────────────────────────────────┘
```

### 用户选择 B（退回重新分析）

将偏离详情作为反馈传递给 requirements-analysis Agent，重新执行 Phase 1。

### 用户选择 A（接受偏离）

记录偏离项，继续 Phase 3。
