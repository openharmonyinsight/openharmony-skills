# 规则013: 枚举值构造不合理

**严重程度**: 中危

**问题描述**: 枚举参数的构造需要在"有效值覆盖"和"非法值覆盖"之间取得平衡。核心原则是：**既要让 fuzz 引擎有机会产生合法枚举值以测试正常路径，也要保留足够的非法值以测试异常处理分支**。

**常见误区**:
- **范围过大**（如 `uint32_t` 0~4294967295）：合法值命中率极低（如枚举值 0~5，命中率仅 0.0000001%），fuzz 几乎总是走非法值分支，无法测试正常逻辑
- **范围过小**（如 `% 8` 限制到 0~7）：虽然提高了合法值命中率，但会遗漏大量非法值处理分支的测试（如 8~255 的异常处理）
- **完全按业务上限取模**（如 `% 6` 限制到 0~5）：**这是最危险的误区**，虽然 100% 命中合法值，但完全丢失了非法值覆盖，无法测试参数校验和错误处理逻辑

**最佳实践**:
1. **使用 `uint8_t` 提取**（范围 0~255），这是枚举测试的黄金标准：
   - 对于常见枚举（有效值 0~N，N < 10），合法值命中率约 2%~10%，既能测试正常路径，又能充分测试异常路径
   - 保留了大量非法值（如 0xFF, 0x80, 0x7F 等特殊值），可测试边界条件和异常处理
   - 不会像 `uint32_t` 那样产生过多无意义的超大值

2. **不要按业务上限取模**：
   - ❌ 不推荐：`mode % 6`（只产生 0~5，完全丢失非法值覆盖）
   - ✅ 推荐：`uint8_t mode = fdp.ConsumeIntegral<uint8_t>();`（产生 0~255，包含合法和非法值）

3. **仅在需要提高合法值密度时适度取模**：
   - 如果业务有复杂的非法值处理逻辑（如不同错误码返回），需要确保有一定概率命中合法值
   - 可以适度取模到一个略大于业务上限的值（如业务上限 5，取模到 16 或 32），既保证合法值密度，又保留非法值覆盖

**核心原则**:
1. 枚举值应使用uint8_t而非大整数类型
2. 不要按业务上限取模，保留非法值覆盖
3. 合法值命中率应保持在合理范围

**错误示例**:
```cpp
// ❌ 使用 uint32_t，合法值命中率极低
void DoSetScreenMode(FuzzedDataProvider& fdp)
{
    uint32_t mode = fdp.ConsumeIntegral<uint32_t>();
    g_instance->SetScreenMode(static_cast<ScreenMode>(mode));
    // ScreenMode 有效值 0~5
    // 合法值命中率: 5/4294967296 ≈ 0.0000001%
    // 99.9999999% 的数据被拦截，无法测试正常逻辑
}

// ❌ 按业务上限取模，完全丢失非法值覆盖
void DoSetScreenMode(FuzzedDataProvider& fdp)
{
    uint8_t mode = fdp.ConsumeIntegral<uint8_t>() % 6;  // 只产生 0~5
    g_instance->SetScreenMode(static_cast<ScreenMode>(mode));
    // 100% 命中合法值，但完全无法测试:
    // - 非法值处理分支（mode > 5 时的错误处理）
    // - 边界值测试（mode = 6, 0xFF, 0x80 等）
    // - 特殊值处理（负数转换、符号位等）
}
```

**正确示例**:
```cpp
// ✅ 使用 uint8_t，平衡合法值和非法值覆盖
void DoSetScreenMode(FuzzedDataProvider& fdp)
{
    // ScreenMode 枚举有效值为 0~5
    uint8_t mode = fdp.ConsumeIntegral<uint8_t>();  // 范围 0~255
    g_instance->SetScreenMode(static_cast<ScreenMode>(mode));
    // 合法值命中率: 5/256 ≈ 2%
    // 0~5: 走正常分支，测试正常逻辑
    // 6~255: 走非法值处理分支，测试参数校验和错误处理
    // 特殊值如 0xFF, 0x80, 0x7F: 测试边界条件和异常处理
}

// ✅ 适度取模，提高合法值密度同时保留非法值覆盖
void DoSetScreenMode(FuzzedDataProvider& fdp)
{
    // ScreenMode 枚举有效值为 0~5
    // 取模到 32，既提高合法值密度，又保留大量非法值
    uint8_t mode = fdp.ConsumeIntegral<uint8_t>() % 32;  // 范围 0~31
    g_instance->SetScreenMode(static_cast<ScreenMode>(mode));
    // 合法值命中率: 5/32 ≈ 15.6%（比不取模的 2% 更高）
    // 0~5: 走正常分支
    // 6~31: 走非法值处理分支（保留了 26 个非法值）
}
```

**检查方法**:
1. 查看枚举定义，了解有效范围
2. 优先使用 `uint8_t` 提取，不要按业务上限取模
3. 仅在合法值命中率过低（如枚举值 > 20 个）时，适度取模到一个略大于业务上限的值
4. 确保用例既能覆盖正常路径，也能覆盖非法值处理路径
5. 检查业务代码中对非法值的处理逻辑，确保 fuzz 有机会测试这些分支

**豁免场景**: 
- API明确设计为接受任意整数并内部处理
- 测试API的输入验证逻辑（故意传非法值）
