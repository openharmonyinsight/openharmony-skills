# 最佳实践

## 何时使用全量覆盖率

### 适用场景
- 需要评估整个部件、模块或子系统的代码覆盖情况
- 定期质量检查（如每周、每月）
- 新功能上线前的全面测试
- 重构后的覆盖度验证

### 优势
- 提供完整的代码覆盖视图
- 便于发现未测试的代码路径
- 有助于识别测试盲点

### 劣势
- 编译和执行时间较长（2-4小时）
- 资源消耗较大（CPU、内存、磁盘）
- 对于大规模项目，生成大量数据

### 建议
- 对于小型部件，推荐使用全量覆盖率
- 对于大型子系统，考虑先使用增量覆盖率，再定期使用全量覆盖率
- 在 CI/CD 流程中，可以将全量覆盖率作为定期任务

---

## 何时使用增量覆盖率

### 适用场景
- 针对代码变更生成覆盖率报告
- PR/合并请求前的验证
- 持续集成中的快速反馈
- 代码审查辅助工具

### 优势
- 执行时间较短（30分钟-2小时）
- 资源消耗较小
- 聚焦于变更部分，更加高效

### 劣势
- 只覆盖变更的代码
- 无法发现未变更部分的覆盖问题
- 不支持子系统级别（仅支持部件级别）

### 建议
- 在开发过程中频繁使用增量覆盖率
- 配合 git diff 自动生成变更文件
- 结合全量覆盖率定期进行全面检查

---

## 跳过编译的注意事项

### 何时可以跳过编译
- 已经完成了一次完整的覆盖率编译
- 没有修改源代码
- 只是想重新生成报告

### 何时不能跳过编译
- 修改了源代码
- 修改了编译配置
- 第一次执行覆盖率测试

### 跳过编译的影响
- 节省编译时间（1-2小时）
- 需要确保之前的编译产物完整
- 可能导致报告不准确

### 建议
- 仅在源代码未修改时使用跳过编译
- 定期（如每周）执行完整编译，确保报告准确性
- 在跳过编译后，如果报告异常，重新执行完整编译

## 设备连接管理

### 单设备场景
- 无需配置 hdc 信息，工具自动查找设备
- 简化配置，快速上手

### 多设备场景
- 必须配置 IP 和端口或序列号
- 确保设备状态稳定
- 避免设备切换导致测试中断

### 设备连接建议
- 使用有线网络连接设备
- 确保网络稳定，避免断连
- 测试前验证设备连接状态
- 定期重启设备，避免资源耗尽

## 性能优化建议

### 编译优化
1. **使用编译缓存**: `--ccache` 启用编译缓存
   - 首次编译后，相同代码的编译时间可减少 60-80%
   - 配置方法：在编译命令中添加 `--ccache`

2. **配置足够的内存和 CPU 资源**
   - 编译阶段内存占用高，建议至少 16GB
   - 多核 CPU 可显著加速编译（16 核比 8 核快 40-60%）

3. **避免在编译高峰期执行**
   - 选择系统负载较低的时间段
   - 关闭不必要的后台进程

### 执行优化
1. **在设备负载较低时执行测试**
   - 避免在设备执行其他任务时运行测试
   - 确保设备有足够的存储空间

2. **使用 SSD 存储提高 I/O 性能**
   - 编译产物和报告存储在 SSD 上可提高 20-30% 的速度
   - 建议使用独立磁盘分区存储报告

3. **合理选择覆盖率类型**
   - 日常开发使用增量覆盖率（1-2小时）
   - 定期使用全量覆盖率（3-5小时）

### 资源监控
1. **监控磁盘空间使用情况**
   - 使用 `df -h` 命令查看磁盘占用
   - 定期清理旧的编译产物和报告

2. **监控内存占用**
   - 使用 `free -h` 命令查看内存使用
   - 如果内存不足，考虑增加 swap 分区

3. **监控 CPU 使用率**
   - 使用 `top` 或 `htop` 命令查看 CPU 占用
   - 如果 CPU 占用过高，检查是否有其他进程干扰

## CI/CD 集成建议

### PR 合并前验证
- 使用增量覆盖率快速验证代码变更
- 设置覆盖率阈值（如覆盖率降低超过 5% 则阻止合并）
- 集成到 PR 检查流程中

### 定期质量检查
- 每周或每月执行全量覆盖率
- 设置定时任务（如 Jenkins Cron、GitHub Actions）
- 生成覆盖率趋势报告

### 持续集成流程
```
代码提交 → PR 创建 → 增量覆盖率验证 → 合并 PR → 全量覆盖率验证
```

### 覆盖率目标设置
- 新代码：覆盖率 ≥ 80%
- 核心模块：覆盖率 ≥ 90%
- 关键路径：覆盖率 ≥ 95%

## 错误处理建议

### 编译错误
**问题**: 编译失败
**排查步骤**:
1. 检查编译日志：`{CODE_ROOT}/out/generic_generic_arm_64only/general_all_phone_standard/build.log`
2. 确认编译环境配置正确
3. 清理编译缓存后重试

**常见原因**:
- 编译环境未正确配置
- 依赖包缺失或版本不匹配
- 源代码语法错误

---

### 执行错误
**问题**: 测试执行失败
**排查步骤**:
1. 检查设备连接状态：`hdc list targets`
2. 确认测试用例存在且正确
3. 检查设备内存和存储空间

**常见原因**:
- 设备连接断开
- 测试用例不存在
- 设备内存不足

---

### 报告生成错误
**问题**: 报告生成失败
**排查步骤**:
1. 检查覆盖率数据文件是否存在
2. 确认 lcov 配置正确
3. 查看详细错误日志

**常见原因**:
- gcda/gcno 文件不匹配
- lcov 配置错误
- 磁盘空间不足

---

## 安全建议

### 配置文件安全
- 不要将 `user-config.json` 提交到代码仓库
- 使用环境变量存储敏感信息
- 定期更换设备密码

**建议配置 .gitignore**:
```
config/user-config.json
coverage_result/
out/
```

---

### 报告安全
- 覆盖率报告可能包含敏感信息
- 根据需要限制报告访问权限
- 及时删除过期的报告

**敏感信息示例**:
- 部分代码实现
- 测试用例名称
- 文件路径信息

---

### 执行安全
- 在隔离的环境中执行测试
- 避免在生产设备上执行测试
- 定期清理临时文件和编译产物

**建议**:
- 使用专用测试设备
- 定期备份数据
- 监控系统资源使用情况

---
