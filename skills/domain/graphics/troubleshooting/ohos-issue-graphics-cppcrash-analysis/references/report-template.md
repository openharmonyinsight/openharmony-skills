# 分析报告模板

## 报告必须包含的内容

1. 元数据总结：设备、进程、崩溃原因等信息
2. 调用栈分析：完整的调用链和关键函数
3. 代码定位：精确的文件名和行号
4. 反汇编分析：崩溃点的指令和寄存器状态
5. 深度数据流分析：寄存器值的来源和变化过程
6. 调用链回溯：从参数来源到崩溃点的完整链条
7. 多线程竞争分析：线程安全问题的分析
8. 内存数据分析：内存布局和异常模式
9. 地址模式匹配：崩溃地址与合法地址的对比分析
10. 根因推断：基于所有证据的合理推断
11. 修复建议：针对性的修复方向和措施

## 根因推断的严谨性

- 必须基于充分的证据，不能凭空猜测
- 必须进行数据流追溯，找到寄存器值异常的根本原因
- 必须进行地址模式匹配，识别内存踩踏特征
- 必须精确定位到代码行
- 必须完整追踪调用链和参数来源
- 必须考虑多种可能性，给出最合理的解释
- 如果证据不足，应明确说明需要进一步分析

## 报告质量要求

- 逻辑清晰，层层递进
- 证据充分，推理严密
- 结论明确，建议具体
- 避免模糊的表述，如"可能是"、"也许是"
- 对于不确定的地方，应明确说明不确定性
- 必须包含代码行号和文件路径

## 分析完成检查清单

在完成分析前，确认一下检查项：

- [ ] 已成功解栈，获取到精确的代码行号
- [ ] 已读取并分析了崩溃点的代码行
- [ ] 已分析反汇编代码，确认崩溃指令
- [ ] 已计算并验证崩溃地址
- [ ] 已追踪寄存器值的来源和变化
- [ ] 已识别异常值的特征（如6b6b开头）
- [ ] 已回溯调用链，找到参数来源
- [ ] 已分析其它线程状态，识别线程安全问题
- [ ] 已分析内存dump，识别异常模式
- [ ] 已进行地址模式匹配，识别踩踏特征
- [ ] 已给出明确的根因推断
- [ ] 已给出具体的修复建议
