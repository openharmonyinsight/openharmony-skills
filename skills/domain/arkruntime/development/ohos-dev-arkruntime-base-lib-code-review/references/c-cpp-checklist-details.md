## 信号安全性

信号安全性（signal-safety）要求在特定执行时机中只可调用异步信号安全函数，否则导致未定义行为（死锁、数据损坏、崩溃）。

**适用时机**：信号处理函数中 | fork 后、exec 前 | 异步回调中 | pthread_atfork 的 child 处理函数中 | vfork 后

**自行开发函数的信号安全规范**：若需在上述时机中被调用，必须同时满足：可重入（不依赖静态/全局变量）、不使用任何锁、不分配/释放堆内存、不使用标准 I/O（如需输出只能用 `write`）、不调用 C++ 异常/虚函数、不依赖其他线程状态。

**常见易错点**：

| # | 易错点 | 说明 |
|---|---|---|
| 1 | fork 后子进程调用非信号安全函数 | fork 只复制调用线程，其他线程持有的锁永远不会释放，子进程中调用持锁函数（printf、fopen、malloc 等）会死锁。正确做法：fork 后 exec 前只调用信号安全函数，或改用 `posix_spawn()` |
| 2 | 信号处理函数入口未保存 errno | `errno` 读写虽是信号安全的，但信号处理函数可能覆盖被中断代码的 errno。必须在入口保存、返回前恢复 |
| 3 | `exit` vs `_exit` | `exit` 会调用 `atexit` 注册的回调并刷新 stdio 缓冲区（非信号安全），信号上下文中只能用 `_exit` |
| 4 | 调用 C++ 编译器隐式生成的非安全代码 | 析构函数、异常处理、RTTI 可能自动插入 `__cxa_atexit`、`__dynamic_cast` 等非信号安全调用 |
| 5 | 信号处理函数中调用业务逻辑 | 被信号中断的业务逻辑可能处于中间不可信状态（全局变量写一半、链表部分更新），读取或修改导致数据损坏 |

## 接口配对使用规范

C/C++ 标准库中许多接口有明确的分配/释放配对要求，必须使用指定的释放函数，不能自行替换。违反配对规范会导致未定义行为、内存损坏或资源泄漏。

**常见配对规范违反案例**：

| # | 正确配对 | 常见错误 | 风险 |
|---|---|---|---|
| 1 | `getaddrinfo` / `freeaddrinfo` | 用 `free()` 释放 `getaddrinfo` 返回的链表 | `getaddrinfo` 返回的是链表结构，`freeaddrinfo` 会遍历释放所有节点，`free()` 只释放头节点，导致内存泄漏；且两者内存分配方式可能不同 |
| 2 | `fopen` / `fclose` | 用 `close()` 关闭 `fopen` 返回的 `FILE*` | `FILE*` 带有 stdio 缓冲区，`close()` 只关闭 fd，缓冲区数据丢失且内存泄漏 |
| 3 | `malloc` / `free`，`new` / `delete`，`new[]` / `delete[]` | `malloc` 分配用 `delete` 释放，或 `new` 分配用 `free` 释放，或 `new[]` 用 `delete`（非 `delete[]`）释放 | `new`/`delete` 会调用构造/析构函数，与 `malloc`/`free` 的内存管理机制不兼容；`new[]`/`delete` 不匹配会导致数组元素析构不完整 |
| 4 | `opendir` / `closedir` | 用 `close()` 关闭 `DIR*` | `DIR*` 是 libc 内部结构，与 fd 不等价，`close()` 无法正确释放内部资源 |
| 5 | `popen` / `pclose` | 用 `fclose()` 或 `close()` 关闭 `popen` 返回的 `FILE*` | `pclose` 会等待子进程结束并获取退出状态，`fclose` 不会，导致僵尸进程 |
| 6 | `tmpfile` / `fclose` | 用 `close()` 或 `unlink` 手动处理 | `tmpfile` 创建的临时文件会在 `fclose` 时自动删除，手动处理可能遗漏 |
| 7 | `regcomp` / `regfree` | 从不调用 `regfree` | 正则表达式编译后分配的内部内存永远不会释放，导致内存泄漏 |
| 8 | `scandir` / `free`（逐个） + `free`（数组） | 只释放数组不释放数组内各元素，或反之 | `scandir` 分配了两层内存：字符串数组 + 每个字符串，需要先逐个 `free` 字符串，再 `free` 数组本身 |
| 9 | `strdup` / `free` | 不释放或用 `delete` 释放 | `strdup` 内部使用 `malloc` 分配内存，必须用 `free` 释放 |
| 10 | `wcstombs` / 自行管理 / `malloc` + `free` | 缓冲区大小计算错误导致截断或溢出 | 需先调用获取所需长度，再分配正确大小的缓冲区 |

**审查要点**：
- 检查所有资源获取/释放是否使用了正确的配对函数
- 特别关注跨层释放（如用 `free` 释放 `new` 分配的内存）
- 关注 C 接口返回的复合结构是否使用了专门的释放函数

## 错误处理与异常捕获

C/C++ 代码中必须正确处理错误，包括返回值检查、errno 判断和 C++ 异常捕获。忽略错误或粗暴处理会导致程序在异常路径上产生不可预测的行为。

**常见问题分类**：

| # | 问题类型 | 说明 |
|---|---|---|
| 1 | 不检查返回值 | 调用函数后不检查返回值（如 `malloc` 返回 NULL、`open` 返回 -1、`pthread_create` 返回非零错误码），直接使用结果导致空指针解引用或操作无效资源 |
| 2 | 不检查 errno | 系统调用失败后不检查 errno，无法区分错误类型，可能做出错误的恢复决策 |
| 3 | 错误处理直接 abort | 判断到错误后直接调用 `abort()` 或 `exit()` 终止进程，而不是尝试优雅恢复或返回错误码。在基础库中更不允许这样做，因为基础库不应擅自终止宿主进程 |
| 4 | 不捕获 std::thread 异常 | `std::thread` 若未 `join` 或 `detach` 则 `terminate`；线程函数内未捕获的异常会导致 `std::terminate`；`std::thread` 构造失败（如资源不足）抛出 `std::system_error` 但调用方未捕获 |
| 5 | 异常规范不当 | 函数声明 `noexcept` 但内部可能抛出异常，导致 `std::terminate`；或应该用 `noexcept` 的移动构造函数/析构函数未标记，影响容器性能 |
| 6 | 析构函数中抛出异常 | 析构函数抛出异常（尤其在栈展开过程中）会导致 `std::terminate`，析构函数必须吞掉异常或终止程序 |
| 7 | 资源清理路径不完整 | 在错误分支中忘记释放已获取的资源（文件描述符、锁、内存等），导致资源泄漏 |

**审查要点**：
- 每个可能失败的系统调用/库函数调用后是否有返回值检查
- `malloc`/`new` 结果是否做了 NULL/异常判断
- `pthread_*` 系列函数的返回值是否做了错误处理（它们不设置 errno，通过返回值返回错误码）
- 错误处理路径中是否正确释放了已获取的资源
- 基础库代码中是否存在 `abort()`/`exit()` 调用（不应有）
- `std::thread` 使用处是否有异常捕获和 join/detach 保证
- 析构函数是否可能抛出异常

## 多线程调用线程不安全接口

部分 C/C++ 标准库接口内部使用静态缓冲区或全局状态，多线程并发调用时会产生数据竞争，导致结果错误或崩溃。

**常见线程不安全接口**：

| # | 接口 | 不安全原因 | 线程安全替代方案 |
|---|---|---|---|
| 1 | `localtime` | 返回指向内部静态 `struct tm` 的指针，并发调用会覆盖彼此结果 | `localtime_r` |
| 2 | `gmtime` | 同 `localtime`，返回指向内部静态缓冲区的指针 | `gmtime_r` |
| 3 | `strtok` | 使用内部静态指针保存解析位置，并发调用互相干扰 | `strtok_r` |
| 4 | `asctime` | 返回指向内部静态字符串的指针 | `asctime_r` |
| 5 | `ctime` | 返回指向内部静态字符串的指针 | `ctime_r` |
| 6 | `rand` | 使用并修改内部种子状态，非原子操作 | `rand_r` 或 C++11 `<random>` |
| 7 | `strerror` | 返回指向内部静态字符串的指针（部分实现） | `strerror_r` 或 `strerror_l` |
| 8 | `getgrgid` / `getgrnam` | 返回指向内部静态 `struct group` 的指针 | `getgrgid_r` / `getgrnam_r` |
| 9 | `getpwuid` / `getpwnam` | 返回指向内部静态 `struct passwd` 的指针 | `getpwuid_r` / `getpwnam_r` |
| 10 | `gethostbyname` / `gethostbyaddr` | 返回指向内部静态 `struct hostent` 的指针 | `getaddrinfo` / `getnameinfo` |
| 11 | `inet_ntoa` | 返回指向内部静态字符串的指针 | `inet_ntop` |

**审查要点**：
- 是否在多线程环境中使用了上述线程不安全接口
- 是否使用了 `_r` 后缀或其他线程安全替代方案
- 封装层是否对外隐藏了线程不安全的内部实现

## setenv/getenv 崩溃风险

`setenv`/`getenv` 因 POSIX 标准和历史实现原因，存在严重的线程安全问题，业界目前没有完美的解决方案。

**问题根因**：

- `getenv` 返回指向 `environ` 内部字符串的指针，不持有任何锁
- `setenv`/`putenv` 修改 `environ` 数组时可能触发 `realloc`，导致 `environ` 整体迁移到新地址
- 若线程 A 正在通过 `getenv` 读取环境变量，线程 B 同时调用 `setenv` 导致 `environ` 迁移，线程 A 持有的指针变为悬空指针，解引用导致崩溃
- 即使不触发 `realloc`，`setenv` 修改已有变量的值时，也可能释放旧值字符串并分配新值，同样导致 `getenv` 返回的指针失效

**风险场景**：

| # | 场景 | 风险 |
|---|---|---|
| 1 | 多线程并发调用 `getenv` 和 `setenv` | `getenv` 返回的指针在 `setenv` 后可能失效，解引用崩溃 |
| 2 | `getenv` 返回值被长期持有 | 持有的指针随时可能因其他线程 `setenv` 而失效 |
| 3 | 调用封装了 `setenv`/`getenv` 的第三方库 | 封装层无法解决底层并发问题，风险依旧存在 |
| 4 | 单线程中 `getenv` 返回值在 `setenv` 后使用 | 即使单线程，`setenv` 也可能使之前 `getenv` 返回的指针失效 |

**审查要点**：
- 是否直接调用了 `setenv`/`getenv`/`putenv`/`unsetenv`
- 是否调用了封装了上述接口的函数（如某些配置框架、日志库的初始化接口）
- 是否有可行的替代方案（如：启动时一次性读取所有环境变量到线程局部存储或自定义 map 中，运行期间不再调用 `setenv`/`getenv`；使用 `secure_getenv` 替代 `getenv` 等）
- 若无法避免使用，是否有并发保护（如全局读写锁保护所有 `setenv`/`getenv` 调用）

## SO 库符号导出控制

编译为 so 动态库时，默认所有符号都可能被导出，暴露内部实现细节，带来以下风险：

- **ABI 耦合**：内部符号被外部依赖后，修改或删除将破坏兼容性
- **符号冲突**：不同 so 库中同名符号会冲突，导致链接到错误实现
- **安全风险**：内部函数被外部直接调用可能绕过安全检查

**符号导出控制方式**：

| # | 方式 | 说明 |
|---|---|---|
| 1 | `__attribute__((visibility("hidden")))` | 隐藏单个符号，不导出到动态符号表 |
| 2 | `-fvisibility=hidden` 编译选项 | 默认隐藏所有符号，仅显式标记为 `default` 的符号导出（推荐方式） |
| 3 | 版本脚本（`.map` / `.version` 文件） | 精确控制导出符号列表，可定义符号版本，推荐用于公共 API 管理 |
| 4 | `__attribute__((visibility("default")))` | 在 `-fvisibility=hidden` 环境下，显式标记需要导出的公共 API |

**审查要点**：
- so 库是否使用了符号导出控制（`-fvisibility=hidden` 或版本脚本）
- 公共 API 是否有明确的导出标记（如 `__attribute__((visibility("default")))` 或导出宏）
- 内部辅助函数是否被误导出（可通过 `nm -D libxxx.so` 或 `readelf -Ws libxxx.so` 检查）
- 版本脚本中的符号列表是否与实际公共 API 一致

## 系统资源与权限限制

Linux 系统中各类资源存在上限，达到限制后系统调用会失败或出现非预期行为，代码必须正确处理这些边界情况。

**常见资源限制与风险**：

| # | 资源 | 限制 | 风险 |
|---|---|---|---|
| 1 | 线程号（TID/PID） | 系统有最大 PID 值（`/proc/sys/kernel/pid_max`，默认 32768 或 4194304），达到上限后 PID 会复用 | 不能用 TID/PID 作为唯一标识符，PID 复用后可能匹配到旧标识符对应的实体（如文件、共享内存）导致误操作 |
| 2 | 文件描述符 | 进程级限制（`ulimit -n`，默认 1024）和系统级限制 | 打开大量文件/套接字后可能耗尽 fd，后续 `open`/`socket`/`pipe` 等调用失败；需检查返回值并做降级处理 |
| 3 | 线程数 | 进程可创建线程数受栈大小和虚拟地址空间限制 | `pthread_create` 返回 EAGAIN，需处理资源不足场景 |
| 4 | 共享内存 | 受 `shmmax`、`shmall` 等参数限制 | `shmget`/`mmap` 可能因限制而失败 |
| 5 | 信号量 | 受 `sem` 值上限限制 | `sem_post`/`sem_wait` 可能返回错误 |
| 6 | 内存 | 受物理内存和 overcommit 策略限制 | `malloc` 可能返回 NULL（或 OOM Killer 杀进程）；`mmap` 可能失败 |
| 7 | 文件系统 | 磁盘空间、inode 数量、文件名长度等限制 | 写文件失败，需处理 `ENOSPC`、`EDQUOT` 等错误 |

**审查要点**：
- 是否将 TID/PID 用作唯一标识符（应使用 UUID 或其他全局唯一标识）
- 资源获取失败时是否有正确的错误处理和降级逻辑
- 长期运行的资源（fd、线程、共享内存）是否有释放机制，防止资源泄漏导致耗尽
- 是否考虑了资源配额限制和权限不足的场景

## POSIX void* 指针解引用风险

POSIX 标准中部分接口的参数或结构体成员类型定义为 `void*`，这是有意为之的抽象设计——标准故意不暴露具体类型，以保留实现的自由度。将 `void*` 解引用为具体类型使用存在 ABI 风险。

**核心风险**：

- **结构体布局变化**：不同 libc 实现或同一 libc 的不同版本中，实际结构体的字段布局、大小、对齐方式可能不同，解引用后访问的字段可能偏移错误或不存在
- **ABI 破坏**：代码依赖了具体的结构体内部实现，一旦 libc 升级改变了内部布局，现有代码将产生内存越界访问或读取错误数据
- **不可移植**：不同平台（glibc、musl、bionic）的内部实现不同，解引用代码无法跨平台工作

**典型案例**：

| # | 接口/结构体 | POSIX 定义 | 常见错误解引用 | 风险说明 |
|---|---|---|---|---|
| 1 | `pthread_key_create` 的 `destructor` 参数接收的 `void*` | `void*` 对应 `pthread_setspecific` 存入的值 | 在 destructor 中将 `void*` 强转为 TLS 相关内部结构体并访问字段 | `void*` 的语义由调用者定义，若调用者存入的是简单指针而非结构体，强转后访问字段会越界 |
| 2 | `dlerror` 返回的 `char*` | 标准仅保证返回错误描述字符串 | 假设字符串内容格式并解析 | 返回的字符串内容因实现而异，解析其格式不可靠 |
| 3 | `dlsym` 返回的 `void*` | 标准定义为 `void*` | 直接将 `void*` 转为函数指针调用 | C 标准不保证 `void*` 和函数指针可互转（虽然 POSIX 追加要求支持），应使用 `*(void**)(&fptr) = dlsym(...)` 模式 |
| 4 | `pthread_attr_getstack` 返回的 `void*` 栈地址 | `void*` 栈基址 | 解引用栈地址访问栈内容 | 栈地址仅用于地址范围判断，不应解引用 |
| 5 | `sigaction` 的 `sa_sigaction` 回调中 `siginfo_t` 的 `si_value.sival_ptr` | `void*` | 假设为特定结构体并解引用 | 该值由 `sigqueue` 发送方设置，类型由发送方决定，接收方不应做假设 |

**审查要点**：
- 是否将 POSIX 接口中定义为 `void*` 的值强制转换为具体结构体类型并解引用访问字段
- 是否依赖了 libc 内部结构体的具体布局（如直接 include 了 libc 内部头文件）
- 是否使用标准提供的接口（getter/setter 函数）来访问 `void*` 背后的数据，而非直接解引用
