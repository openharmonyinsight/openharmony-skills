## Q1: 生成的测试用例无法编译

### 可能原因

1. hypium 导入不正确
2. ArkTS 语法类型不匹配
3. 测试文件路径错误
4. @tc 注释块格式错误

### 解决方案

1. **检查 hypium 导入是否正确**：
   ```typescript
   import {describe, beforeAll, beforeEach, afterEach, afterAll, it, expect, TestType, Size, Level} from '@ohos/hypium';

   ```

2. **检查 ArkTS 语法类型是否匹配**：
   - 读取 `.d.ts` 文件中的 `@since` 标签
   - 读取 `build-profile.json5` 中的 `arkTSVersion` 字段
   - 确保工程类型与 API 类型兼容

3. **检查测试文件路径**：
   - ArkTS 动态测试套（arkts-dyn）：`entry/src/ohosTest/ets/test/` 目录
   - ArkTS 静态测试套（arkts-sta）：`entry/src/main/src/test/` 目录
   - 确保文件命名符合规范

4. **检查 @tc 注释块格式**：
   ```typescript
   /**
    * @tc.name onClickNormal001
    * @tc.number SUB_ARKUI_COMPONENT_ONCLICK_PARAM_001
    * @tc.desc 测试 Component 的 onClick 方法 - 正常点击场景.
    * @tc.type FUNCTION
    * @tc.size MEDIUMTEST
    * @tc.level LEVEL3
    */
   ```

5. **查看编译错误日志**：
   - 参考 `modules/L4_Build/linux_compile_troubleshooting.md`

## Q2: 测试用例命名不符合规范

### 可能原因

1. 使用了大写下划线命名
2. 包含特殊标点符号
3. @tc.name 与 it() 第一个参数不一致

### 解决方案

1. **确保使用小驼峰命名（camelCase）**：
   - ✅ 正确示例：`onClickNormal001`
   - ✅ 正确示例：`uitestOnText401`
   - ❌ 错误示例：`ON_CLICK_NORMAL_001`
   - ❌ 错误示例：`test[MethodName].Param.0001`

2. **确保 @tc.name 与 it() 第一个参数完全一致**：
   ```typescript
   /**
    * @tc.name onClickNormal001
    */
   it('onClickNormal001', Level.LEVEL3, () => {
     // 测试代码
   });
   ```

3. **参考命名规范**：
   - 查看 `references/subsystems/_common.md` 的命名规范章节

## Q3: 测试设计文档与测试用例不一致

### 可能原因

1. 测试用例修改后未同步更新设计文档
2. 设计文档版本号未递增

2. **检查设计文档的版本历史**，确保版本号递增：
   ```markdown
   | 版本 | 日期 | 变更内容 | 作者 |
   |------|------|---------|------|
   | 1.0 | 2026-02-10 | 初始版本 | oh-xts-generator-template |
   | 1.1 | 2026-02-11 | 修改场景3的预期结果 | User |
   ```

3. **更新变更记录**，记录修改内容

4. **参考更新机制**：
   - 查看 `modules/L3_Generation/design_doc_generator.md`

## Q4: Linux 环境编译失败

### 可能原因

1. 使用了 `hvigorw` 命令（不适用于 Linux）
2. 编译环境未正确配置
3. BUILD.gn 编译目标识别错误
4. 预编译清理不彻底

### 解决方案

1. **必须使用 `build.sh` 脚本编译**，不要使用 `hvigorw`：
   ```bash
   ./test/xts/acts/build.sh product_name=rk3568 system_size=standard suite=test_name
   ```

2. **检查编译环境配置**：
   - 参考环境准备指南：`modules/L4_Build/linux_compile_env_setup.md`
   - 确保工具链安装正确
   - 确保 SDK 下载完整

3. **执行预编译清理**：
   ```bash
   rm -rf build
   rm -rf out
   ```
   - 参考清理指南：`modules/L4_Build/linux_prebuild_cleanup.md`

5. **查看编译错误日志**：
   - 参考问题排查指南：`modules/L4_Build/linux_compile_troubleshooting.md`

在以下情况下需要调用 arkts-static-spec 技能：

1. **静态语法校验**
   - 当需要进行 ArkTS 静态语言语法规范检查时
   - 当需要验证类型系统正确性时

2. **编译问题分析**
   - 当遇到静态编译错误时
   - 当需要理解类型不匹配问题时

3. **代码规范检查**
   - 当需要确保代码符合 ArkTS 静态规范时
   - 当进行 TypeScript 迁移时

### 调用方式

```bash
请使用 arkts-static-spec 进行语法规范校验：
- 语法规则检查
- 类型系统验证
- 编译问题分析
```

### 注意事项

- 确保已安装 arkts-static-spec 技能
- 遵循该技能的使用原则和限制
- 不要在 oh-xts-generator-template 中重复实现静态语法检查功能

## Q10: 如何使用覆盖率报告驱动模式

### 什么是覆盖率报告驱动模式？

覆盖率报告驱动模式是一种高效的工作流程，当用户提供测试覆盖率报告时，系统会直接基于报告中列出的缺失测试项生成测试用例，无需进行覆盖率分析。

### 何时使用此模式？

在以下情况推荐使用覆盖率报告驱动模式：

1. **已有覆盖率报告**：您已经通过其他工具生成了测试覆盖率报告
2. **快速补充测试**：需要快速补充缺失的测试用例
3. **精准生成**：希望严格按照覆盖率报告中指定的缺失项生成测试
4. **大型项目**：对于大型 API 集合，避免重复分析

### 工作流程差异

| 对比项 | 标准流程 | 覆盖率报告驱动流程 |
|--------|---------|-------------------|
| 步骤3 | 测试覆盖分析（扫描现有测试） | 解析覆盖率报告（提取缺失项） |
| 效率 | 中等（需要完整分析） | 高（直接生成） |
| 精准度 | 基于分析结果 | 基于报告明确指出的缺失项 |
| 适用场景 | 新测试生成、完整分析 | 补充缺失测试、快速迭代 |

### 如何使用？

#### 方式1：在初始请求中提供覆盖率报告

```bash
请使用 oh-xts-generator-template 生成测试用例，子系统：XXX

覆盖率报告：
```
[在此粘贴覆盖率报告内容]
```

目标 API：[API名称列表，可选]
```

#### 方式2：在对话中提供覆盖率报告

```bash
这是覆盖率报告：
```
[在此粘贴覆盖率报告内容]
```

请根据覆盖率报告中缺失的内容补充测试用例
```

### 覆盖率报告格式要求

覆盖率报告应包含以下关键信息：

1. **未覆盖的 API 名称**
   - 明确列出需要测试的 API
   - 包含命名空间或类名（如适用）

2. **缺失的参数组合**
   - 指出哪些参数组合未被测试
   - 包括边界值和特殊值

3. **缺失的测试场景**
   - PARAM：参数测试
   - ERROR：错误码测试
   - RETURN：返回值测试
   - BOUNDARY：边界值测试

4. **测试类型标识**
   - 明确缺失的测试类型
   - 按优先级排序（如有）

### 示例覆盖率报告

```
覆盖率报告：Component.onClick API

未覆盖的测试项：
1. onClickNormal001 - 正常点击场景（PARAM）
2. onClickError401 - 无权限场景（ERROR）
3. onClickBoundary001 - 多次快速点击（BOUNDARY）
4. onClickReturn001 - 返回值验证（RETURN）

缺失参数组合：
- onClickEvent 对象为 null
- onClickEvent 对象属性缺失
- onClickEvent 对象属性类型错误
```

### 系统行为

当检测到覆盖率报告时，系统会：

1. ✅ **自动识别**：识别用户提供的是覆盖率报告
2. ✅ **跳过分析**：跳过标准流程中的步骤3（测试覆盖分析）
3. ✅ **提取信息**：从报告中提取缺失的 API、参数组合、测试场景
4. ✅ **代码风格**：仅扫描现有测试文件以获取代码风格（不进行覆盖率分析）
5. ✅ **生成用例**：直接基于缺失项生成测试用例
6. ✅ **保持规范**：严格遵循子系统配置和 API 声明定义

### 优势

- **节省时间**：跳过完整的覆盖率分析，显著缩短生成时间
- **精准补充**：严格按照报告中列出的缺失项生成，避免冗余
- **可控性强**：用户可以明确控制需要补充哪些测试
- **迭代快速**：特别适合持续补充测试的场景

### 注意事项

1. **报告准确性**：确保提供的覆盖率报告准确无误
2. **格式规范**：报告内容应清晰、结构化
3. **API 声明**：生成的用例仍需严格遵循 `.d.ts` 文件中的 API 声明
4. **代码风格**：会从现有测试文件中提取代码风格，保持一致性

### 与标准流程的兼容性

- 两种流程共享相同的后续步骤（API 解析、生成、验证等）
- 生成的测试用例质量和规范完全一致
- 可以在标准流程和覆盖率报告驱动流程之间自由切换

## Q11: 为什么格式化和验证步骤经常被漏掉？

### 问题描述

在使用 oh-xts-generator-template 生成测试用例时，经常漏掉"格式化和验证"步骤，导致后续编译失败或测试用例质量不佳。

### 根本原因分析

#### 1. **步骤描述不够突出**
- 格式化和验证只是众多步骤之一，没有强调其**强制性**
- 缺乏视觉标记（如 ⚠️、✅）来提醒此步骤的重要性
- 在长流程中容易被忽略

#### 2. **依赖关系复杂**
- 需要调用 `validation-agent subagent`
- 静态语法校验还需调用 `arkts-static-spec` 技能
- 条件分支多（动态 vs 静态），增加执行复杂度

#### 3. **缺乏检查点**
- 没有明确的机制来验证此步骤是否完成
- 缺乏自动化检查（如代码质量门禁）
- 没有强制性的完成条件

#### 4. **用户认知偏差**
- 用户可能误以为"编译验证"会自动发现所有问题
- 不清楚"格式化和验证"与"编译验证"的区别：
  - **格式化和验证**：检查语法规范、命名规范、注释规范、断言方法
  - **编译验证**：检查编译环境、编译目标、依赖关系
- 认为此步骤是"可选的优化步骤"

#### 5. **流程不够连贯**
- 步骤间缺乏明确的触发条件
- 没有"完成条件"说明
- 从"生成测试用例"到"注册测试套"可能直接跳过验证

### 解决方案

#### 方案 1：明确步骤的强制性（已实施）

**措施**：
- 在核心功能中添加 ⚠️ 标记
- 在工作流程中明确标注"⚠️ 格式化和验证（强制步骤）"
- 添加"完成条件"和"检查项"清单

**效果**：
- 视觉上突出此步骤的重要性
- 明确告知用户此步骤不可跳过
- 提供明确的完成标准

#### 方案 2：简化依赖关系

**措施**：
- 将 validation-agent 的调用方式简化
- 静态语法校验作为可选补充（仅在必要时调用）
- 提供一键式验证命令

**示例**：
```bash
# 简化前的命令
请使用 validation-agent 检查生成的测试用例：
- ArkTS 动态语法规范检查
- @tc 注释块完整性检查
- 断言方法正确性验证

# 简化后的命令
请执行格式化和验证步骤（强制）：
- 检查语法规范和命名规范
- 验证断言方法正确性
- 确保符合 XTS 规范
```

#### 方案 3：添加自动化检查点

**措施**：
- 在技能内部添加自动检查机制
- 在输出文件列表时自动验证：
  - 是否存在语法错误
  - 是否符合命名规范
  - @tc 注释块是否完整
- 提供检查报告

**示例输出**：
```
✅ 格式化和验证检查报告：
- ArkTS 语法：通过
- @tc 注释块：通过
- 命名规范：通过
- 断言方法：通过

⚠️ 发现问题：
- 文件 Component.onClick.test.ets:45：缺少 @tc.level 字段
- 文件 Component.onClick.test.ets:120：测试用例命名不符合规范（应使用小驼峰）
```

#### 方案 4：用户教育和文档优化

**措施**：
- 在 SKILL.md 重要注意事项中添加专门的说明（已实施）
- 在故障排除文档中添加此 FAQ（本文档）
- 提供最佳实践示例

**教育内容**：
- 解释为什么此步骤不可跳过
- 说明不执行此步骤的后果
- 提供完整执行流程示例

#### 方案 5：流程改进

**措施**：
- 在"生成测试用例"步骤中嵌入初步验证
- 将"格式化和验证"作为独立的检查点
- 在"编译验证"前强制确认验证结果

**改进后的流程**：
```
4. 生成测试用例和测试设计文档
   ├─ 生成测试用例
   ├─ 初步检查：语法、命名、注释
   └─ 生成测试设计文档

5. 注册测试套
    ├─ 在 List.test.ets 中注册新增测试文件
    ├─ 添加 import 语句
    └─ 在 testsuite() 函数中调用

6. ⚠️ 格式化和验证（强制步骤）
    ├─ 深度检查：语法规范、断言方法
    ├─ 检查注册信息的正确性
    ├─ 生成验证报告
    └─ 确认通过后才继续
```

### 最佳实践

#### ✅ 推荐做法

1. **始终执行格式化和验证步骤**
    ```bash
    # 正确流程
    生成测试用例 → 注册测试套 → 格式化和验证 → 编译验证
    ```

2. **使用检查清单**
   ```markdown
   格式化和验证检查清单：
   - [ ] 语法错误已修复
   - [ ] @tc 注释块完整
   - [ ] hypium 导入正确
   - [ ] 命名符合规范
   - [ ] 断言方法正确
   ```

3. **阅读验证报告**
   - 仔细阅读 validation-agent 的输出
   - 按照报告中的建议修复问题
   - 确保所有检查项都通过

4. **在编译前确认**
   ```bash
   # 编译前确认
   请确认：格式化和验证步骤已完成 ✓
   执行编译命令...
   ```

#### ❌ 错误做法

1. **直接跳过验证步骤**
    ```bash
    # 错误流程
    生成测试用例 → 注册测试套 → 直接编译验证 ❌
    ```

2. **认为编译会自动发现问题**
   ```bash
   # 错误认知
   "编译失败时会告诉我问题在哪" ❌
   # 实际：编译只能发现部分问题，很多规范问题编译不报错
   ```

3. **忽略验证报告中的警告**
   ```bash
   # 错误做法
   "警告可以忽略，能编译就行" ❌
   # 实际：警告可能导致测试执行失败或结果不准确
   ```

4. **不完整的验证**
   ```bash
   # 错误做法
   只检查语法，不检查命名和注释 ❌
   # 实际：所有检查项都是强制性的
   ```

### 技术对比

| 对比项 | 格式化和验证 | 编译验证 |
|--------|------------|---------|
| **检查内容** | 语法规范、命名规范、注释规范、断言方法、注册信息 | 编译环境、编译目标、依赖关系 |
| **发现的问题** | 规范性问题、代码质量问题、注册错误 | 编译错误、链接错误 |
| **执行时机** | 注册测试套后 | 格式化和验证后 |
| **强制程度** | ⚠️ 强制步骤 | ⚠️ 强制步骤 |
| **可跳过性** | ❌ 不可跳过 | ❌ 不可跳过 |
| **相互依赖** | 编译验证的前置条件、注册测试套的后置步骤 | 格式化和验证的后置步骤 |

### 总结

**格式化和验证步骤的核心价值**：
- ✅ 确保测试用例符合 XTS 规范
- ✅ 提高测试用例质量和可维护性
- ✅ 避免编译阶段才发现问题
- ✅ 提高测试执行成功率

**关键要点**：
1. 此步骤是**强制性的**，不可跳过
2. 此步骤在注册测试套**之后**执行，检查注册信息的正确性
3. 此步骤与编译验证**相互独立**，各有侧重
4. 必须按照**检查清单**逐项确认
5. 仔细阅读**验证报告**，修复所有问题
6. 确认通过后才继续后续步骤
