# OpenHarmony C++ 规范到 Skill 条目筛选

## 目标
本文件把 OpenHarmony C++ 编码规范压缩为后续 skill 的候选规则。原始规范通常位于 OpenHarmony 文档仓的 `zh-cn/contribute/OpenHarmony-cpp-coding-style-guide.md`。

新的筛选口径比上一版更严格，只保留：

1. OpenHarmony 特有或非常明显的项目偏好
2. 高区分度、会明显改变代码结构/接口/所有权模型的规则
3. 不能主要依赖 `clang-format`、`clang-tidy`、编译器告警来完成的约束

明确压缩或忽略：

1. 纯格式、排版、空格、换行、缩进
2. 普通现代 C++ 常识
3. 更适合静态检查器/编译器/代码 review 兜底的通用规则

## 方法说明

这份文档不再追求“逐条都给一个适合写进 skill 的结论”，而是改成三层：

- `保留为 skill 核心条目`
- `保留为简短补充说明`
- `忽略`

## 保留为 Skill 核心条目

这些条目应直接进入后续 `SKILL.md` 主体。

| 条目 | 原因 | 处理策略 |
|---|---|---|
| 规则2.2.1 C++文件以`.cpp`结尾，头文件以`.h`结尾 | 明确项目约定，不是所有仓库都会这样做。 | 新增文件默认使用 `.h/.cpp`。 |
| 规则2.2.2 文件名和类名保持一致，文件名用下划线小写 | OpenHarmony 命名习惯明确，模型常会默认用 `CamelCase` 文件名。 | 主类型对应文件名用 `snake_case`。 |
| 规则2.5.1 全局变量加 `g_` 前缀 | 具体命名约定，区分度高。 | 若必须使用全局变量，统一 `g_` 前缀。 |
| 规则2.5.2 类成员变量小驼峰加后下划线 | 这是 OpenHarmony 风格里很显眼的命名要求。 | 类成员使用 `fooBar_`；struct/union 成员不加 `_`。 |
| 规则3.13.2 避免使用宏 | 明显项目偏好，且宏滥用会明显改变代码风格。 | 默认不用宏，优先常量、枚举、函数。 |
| 规则3.13.3 禁止用宏表示常量 | 明确硬约束。 | 不写常量宏。 |
| 规则3.13.4 禁止使用函数式宏 | 明确硬约束。 | 默认用函数/内联函数替代；必要日志场景例外。 |
| 规则4.1 文件头注释必须包含版权许可 | 项目特定要求，模型不会自然补齐。 | 新建 `.h/.cpp` 文件时补 Apache 2.0 版权头。 |
| 规则4.3.1 public 函数必须写函数头注释 | 项目文档化要求，区分度高。 | 对外接口写用途、约束、返回值、内存约定等。 |
| 规则4.3.2 禁止空有格式的函数头注释 | 防止生成模板废话注释。 | 只写签名无法表达的信息。 |
| 规则5.2.1 禁止头文件循环依赖 | 更偏结构设计约束，不是格式器问题。 | 设计接口时优先消除循环依赖。 |
| 规则5.2.2 头文件必须用`#define`保护，不用`#pragma once` | 很强的项目偏好。 | 头文件统一 include guard。 |
| 规则5.2.3 禁止通过 extern 声明引用外部接口/变量 | 很具体，模型图省事时容易写错。 | 跨文件接口必须通过头文件声明引入。 |
| 规则5.2.4 禁止在`extern \"C\"`中包含头文件 | 典型易错点，非通用模型默认输出。 | 不把 `#include` 放进 `extern \"C\"`。 |
| 建议5.2.1 尽量避免前置声明而改用`#include` | 与很多现代 C++ 习惯不同，属于项目偏好。 | 默认优先真实 `#include`，少用前置声明。 |
| 建议6.1.1 cpp 内部符号用匿名 namespace 或 static | OpenHarmony 风格偏好明显。 | `.cpp` 内部仅文件可见符号放匿名 namespace；不要放进 `.h`。 |
| 规则6.1.1 不要在头文件或`#include`前 using namespace | 高区分度约束，容易污染接口边界。 | 头文件禁 `using namespace`；源文件中也不要在 `#include` 前写。 |
| 建议6.2.1 全局函数优先放命名空间 | 项目代码组织偏好。 | 避免“工具类+静态方法”替代命名空间。 |
| 建议6.3.1 全局常量优先放命名空间 | 项目代码组织偏好。 | 全局常量默认放命名空间。 |
| 规则7.1.3 不需要复制/移动时必须显式禁止 | 会显著影响类语义设计。 | 不用的拷贝/移动能力显式禁用，不用 `DISALLOW_*` 宏。 |
| 规则7.1.7 多态基类的拷贝/移动必须非 public 或 delete | 具体且高风险，影响继承体系设计。 | 多态基类默认禁用 public 拷贝/移动。 |
| 规则7.2.1 基类析构函数应为 virtual，不准备继承的类用 final | 明显影响类设计边界。 | 可继承基类虚析构；无继承意图类显式 `final`。 |
| 规则7.2.2 虚函数禁止默认参数 | 具体、易忽略、非格式问题。 | 虚函数不声明默认实参。 |
| 规则7.2.3 禁止重新定义继承来的非虚函数 | 明确的面向对象约束。 | 需要多态时修改基类设计，不做同名遮蔽。 |
| 规则9.5.1 不要保存`std::string::c_str()`返回的指针 | 具体易错点，普通检查器不一定覆盖到位。 | 只在调用点临时取 `c_str()`。 |
| 规则9.8.1 OpenHarmony项目中禁止泛型编程 | 这是最强的项目特有偏好之一。 | 默认不要自定义模板/泛型；仅少数允许部件例外。 |
| 规则10.2.1 单一所有权成员/单例优先原始指针而不是智能指针 | 与常见现代 C++ 默认风格不同，区分度很高。 | 类成员/单例等独占生命周期场景默认原始指针。 |
| 规则10.3.1 非局部 lambda 避免引用捕获 | 高风险且不主要是格式/静态检查问题。 | 返回、跨线程、堆上存储的 lambda 不要引用捕获局部变量。 |
| 建议10.3.2 捕获 this 时显式捕获所有变量 | 明显改变 lambda 写法，区分度高。 | 涉及 `this` 时显式列出捕获。 |
| 建议10.3.3 避免默认捕获模式 | 与上条配套。 | 默认不用 `[=]` / `[&]`。 |
| 建议10.4.1 非所有权场景参数用`T*`/`T&`，不用智能指针 | 与 10.2.1 一起构成强烈的 OpenHarmony 接口偏好。 | 不表达所有权的接口不要传智能指针。 |

## 保留为简短补充说明

| 条目 | 处理方式 |
|---|---|

## 结论

按这版更严格的标准，后续 `SKILL.md` 应以“项目特有约束”而不是“通用 C++ 指南摘要”为目标。

建议后续 skill 主体控制在 12-18 条规则内，避免变成泛化 C++ 教程。
