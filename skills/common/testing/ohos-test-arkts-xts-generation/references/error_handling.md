# 错误分类与处理指南

> ohos-test-arkts-xts-generation 生成过程中的错误分类、处理和重试策略

## 1. 错误分类

### Level 1：自动修复（无需用户干预）

| 错误类型 | 触发场景 | 处理策略 |
|----------|----------|----------|
| ArkTS 语法错误 | 静态语法违规 | 调用 arkts-static-spec 校验，自动修复 |
| @tc 注释块缺失 | 验证阶段发现 | 按模板自动补全 |
| hypium 导入错误 | 导入路径不正确 | 按规范自动修正 |
| 命名不规范 | camelCase 违规 | 自动转换为 camelCase |
| 断言方法错误 | 使用了 assertNotX | 替换为正确的断言方法 |

### Level 2：重试修复（自动重试后可能需用户确认）

| 错误类型 | 触发场景 | 处理策略 |
|----------|----------|----------|
| 编译语法错误 | build.sh 编译失败 | 自动修复，最多重试 3 次 |
| @since 标签解析失败 | API 语法类型未知 | 降级为 HYBRID 类型 |
| .d.ts 文件过大 | 读取超时 | 分块读取（200 行/块） |

### Level 3：需用户确认

| 错误类型 | 触发场景 | 处理策略 |
|----------|----------|----------|
| 编译配置错误 | BUILD.gn 配置问题 | 展示错误信息，等待用户确认 |
| 子系统配置冲突 | 配置规则矛盾 | 展示矛盾内容，等待用户裁定 |
| API 签名歧义 | .d.ts 与文档不一致 | 展示证据和选项，等待用户选择 |
| 覆盖率报告格式异常 | 无法解析报告 | 请求用户提供正确格式的报告 |

### Level 4：任务终止

| 错误类型 | 触发场景 | 处理策略 |
|----------|----------|----------|
| OH_ROOT 不存在 | 路径配置错误 | 终止，提示修正 .oh-xts-config.json |
| .d.ts 文件不存在 | API 声明文件缺失 | 终止，提示检查路径 |
| 无任何测试代码 | 完全无法参考 | 继续但标注"无测试参考" |

## 2. 编译错误重试策略

### 语法错误（自动修复）

```
编译失败 → 提取错误信息 → 判断错误类型
  ├─ ArkTS 语法错误 → 调用 arkts-static-spec 校验 → 修复 → 重新编译
  ├─ hypium 导入错误 → 修正导入语句 → 重新编译
  ├─ 断言方法错误 → 替换断言 → 重新编译
  └─ 其他语法错误 → 展示错误 → 尝试修复 → 重新编译

最多重试 3 次。3 次后仍失败 → 提交给用户确认。
```

> **重试约束**：
> 1. 每次重试必须采用不同的修复策略，禁止用相同修复重复尝试（相同修复重复尝试无法获得新信息，浪费编译时间）
> 2. 每次重试必须保留上一次的完整错误信息和修复内容记录（保留历史才能让用户看到完整的排查链路）
> 3. 3 次全部失败后，向用户提供完整的错误-修复历史（不是只展示最后一次）
> 4. 修复过程中不得扩散修改范围（不应修改非本次生成的新文件）（扩散修改可能破坏已通过的测试，引入新的回归问题）

### 配置错误（需用户确认）

```
编译失败 → 提取错误信息 → 判断为配置错误
  → 展示完整错误信息
  → 列出可能的解决方案
  → 等待用户确认
```

> **信息保留要求**：自动修复过程中的原始错误信息不得丢弃。每次修复应记录：
> - 原始错误信息（完整错误栈）
> - 修复方案和修复内容
> - 修复后的编译结果
> 最终在 Phase 10 输出中包含修复历史摘要。

## 3. 验证阶段错误处理

validation-agent 检查发现问题时：

| 检查项 | 失败处理 |
|--------|----------|
| ArkTS 语法 | 自动修复后重新检查 |
| @tc 注释块 | 自动补全后重新检查 |
| hypium 导入 | 自动修正后重新检查 |
| 命名规范 | 自动转换后重新检查 |
| 断言方法 | 替换后重新检查 |

所有自动修复完成后，如果仍有未通过的检查项，提交给用户确认。
