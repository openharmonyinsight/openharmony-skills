# 开合连续性指南（官方 UX 标准 4.1 对齐）

## 官方基线

来源：华为开发者联盟《折叠屏应用 UX 体验标准》4.1 开合连续性。
参考链接：https://developer.huawei.com/consumer/cn/doc/design-guides/ux-guidelines-foldable-screen-0000001807866557

核心要求：应用在设备折叠/展开后，不应出现操作步骤增加或体验变复杂。典型风险包括：
- 页面切换到其他页面
- 页面滚动位置发生偏移
- 输入内容丢失
- 图片变模糊
- 播放进度变化

## 工程落地要求

1. 状态连续：折叠/展开前后的业务状态保持同一语义，不额外插入“恢复步骤”。
2. 视口连续：滚动容器按目标布局重算后再恢复锚点，禁止基于过期视口直接回填偏移。
3. 输入连续：输入草稿、编辑态、光标位置可恢复，避免开合后重置。
4. 媒体连续：播放时间轴以前台媒体时钟为准，避免重复补偿导致进度漂移。
5. 画质连续：预览降级策略必须可回收，开合完成后恢复到目标清晰度。

## 关键通用修改方案（官方导向）

> 对齐官方 4.1「开合连续性」，目标是“开合后不增加用户步骤、不改变任务语义”。

### 方案一：状态快照 + 单一状态源

- 在 fold status / display mode 变化入口，先采集快照，再切换布局。
- 快照至少包含：`播放进度`、`播放/暂停状态`、`滚动锚点`、`输入态`。
- 所有布局分支（展开/折叠/悬停）复用同一套状态读写函数，避免多套逻辑漂移。

### 方案二：双触发恢复 + 稳定窗口

- 恢复触发点至少两类：
  - 折展状态变化事件（防止未触发 prepared 时丢恢复）
  - 组件 prepared（防止重建后首帧状态不一致）
- 在折展稳定窗口内允许幂等重放恢复，窗口结束后再清理恢复标记。
- 推荐思路：`立即恢复一次` + `稳定窗口内按需补偿`，而不是只恢复一次。

### 方案三：播放态与暂停态分治

- 暂停态：`暂停状态优先级高于 autoplay`，折展期间若被系统拉起播放应立即回拉到暂停。
- 播放态：避免高频强制 seek，优先保持“继续播放”，时间推进交给播放器时钟。
- 这样可同时避免两类问题：`暂停后变播放`、`播放中反复跳秒`。

### 方案四：恢复链路去抖与可回收

- 恢复任务必须支持去抖、可重入、可取消（页面消失或状态稳定后回收）。
- 禁止“消费一次即失效”的恢复 token 设计，否则在多次重建时容易丢状态。

### 方案五：连续性验证最小集合

- 暂停态：`展开 -> 折叠/悬停 -> 展开`，两次采样间隔 3s，进度应保持不变。
- 播放态：在约 3s 位置执行同样折展链路，不应回到 0s 重播；时间轴应连续推进（短视频循环场景按循环边界判定，不算回 0 缺陷）。
- 连续折展多轮（如 10 次），确认不出现累计偏移、输入丢失、清晰度不回升。

## 媒体连续性补充规则（问题复盘沉淀）

### 规则一：悬停链路与单屏链路必须分开验收

- 媒体连续性必须至少覆盖两条链路：
  - `展开 -> 悬停(z) -> 展开(y)`
  - `展开 -> 单屏(p) -> 展开(y)`
- 两条链路都通过，才可判定“开合连续性通过”。
- 禁止只用单屏链路结论替代悬停链路结论。

### 规则二：恢复状态不得按固定短超时提前清空

- 禁止“捕获快照后固定 N ms 清空 pending restore”。
- 恢复完成条件应为：
  - 已观察到“捕获快照之后的新 prepared”，或
  - 达到更长的兜底等待窗口后再结束恢复。
- 目标：避免悬停路径重建更慢时，恢复状态先失效导致进度回 0。

### 规则三：autoplay 不能覆盖业务暂停态

- 当存在 pending restore 时，禁止 autoplay 抢先拉起播放。
- 暂停态恢复优先级高于 autoplay，确保“暂停后折展仍保持暂停”。
- 用户点击播放一次后必须直接进入播放态，不能出现“闪一下又回暂停”的二次点击问题。

### 规则四：恢复完成前，恢复任务要可重放

- 恢复任务需具备：可重入、可去抖、可取消。
- 在折展稳定前允许幂等重放恢复；稳定后再清理恢复标记。
- 不能使用“消费一次即失效”的恢复 token 方案。

### 规则五：媒体回归门禁用例（P0 必测）

- 用例 A（暂停态保持）：
  - 在约 `3s` 暂停，执行 `y->z->y` 与 `y->p->y`。
  - 期望：折展后仍为暂停，进度保持在 `3s` 附近（容忍采样误差）。
- 用例 B（单击播放可恢复）：
  - 在用例 A 结束后点击一次播放。
  - 期望：进度单调前进，不出现“回 3s 再暂停”。
- 用例 C（播放态连续）：
  - 在约 `3s` 播放中执行 `y->z->y` 与 `y->p->y`。
  - 期望：时间轴连续推进；若素材循环，按循环边界解释，不误判为重置。

## 验收与测试

- 官方测试方法：设备展开/折叠，观察开合转换过程和页面布局。
- 官方判定标准：开合过程流畅不卡顿，界面布局正常，操作交互正常。
- 官方标准等级：必须。

建议增加以下专项用例：
- 连续折叠/展开 10 次，检查滚动锚点漂移量是否累计。
- 输入中途折展，检查输入内容、光标、候选词上下文是否保持。
- 视频播放中折展，检查时间轴是否单调连续且无明显跳变。
- 图片列表折展，检查清晰度是否在稳定态回升。

## 常见根因清单

- 旧恢复链路用上一帧高度恢复视口，导致偏移累加。
- 折展后渲染预算收缩未回补，导致可见内容减少或文本截断。
- 预览降级参数累积未清零，导致持续模糊。
- 时间线补偿叠加系统回调，导致播放进度不一致。
- 仅依赖 prepared 做恢复，导致“事件已发生但未重建”路径丢状态。
- autoplay 与业务暂停态冲突，导致“暂停 -> 折展 -> 自动播放”回归。
