# 折展问题修复场景库（通用）

## 目的

统一沉淀折展问题修复方法，所有修复场景使用同一结构输出：
- 问题描述
- 根因分析
- 通用修复方案

本文件只保留通用方案，不绑定特定页面、业务或产品案例。

## 使用方式

1. 先识别问题类型，定位到对应场景。
2. 按“问题描述 -> 根因分析 -> 通用修复方案”执行。
3. 修复后按统一验证矩阵回归。

## 场景 A：悬停态布局未生效

### 问题描述
- 折叠到半折叠后仍沿用常态单屏布局。
- 关键内容或操作区跨折痕，导致可用性下降。

### 根因分析
- 缺少 `foldStatusChange` / `foldDisplayModeChange` 监听入口。
- 悬停态与常态布局分支未拆分，状态驱动链路不完整。

### 通用修复方案
- 建立统一状态入口，确保折展变化能触发布局刷新。
- 为悬停态单独提供分屏结构（上展示、下交互），常态保留原断点策略。
- 悬停态与常态复用同一业务状态机，只变更布局承载位置。

## 场景 B：方向反复切换或偶发反向

### 问题描述
- 进入悬停后方向来回翻转，界面抖动。
- 折展过程中偶发方向与页面语义不一致。

### 根因分析
- 在尺寸变化或多处回调中重复执行方向决策。
- 方向决策优先级错误（先猜几何、后看语义）。

### 通用修复方案
- 进入悬停时只执行一次方向决策，悬停中禁止重复选方向。
- 悬停中仅允许重算分界线和区域尺寸，不改方向状态。
- 退出悬停或页面离开时恢复 `AUTO_ROTATION`，并回收监听。

## 场景 B-2：竖折痕左右折叠时未按语义翻转 90 度

### 问题描述
- 页面要求“上展示下操作”，但竖折痕进入悬停后仍表现为未翻转方向。
- 同一页面在不同折展路径中，方向有时正确有时回退。

### 根因分析
- 方向决策过度依赖当前 `rootWidth/rootHeight`，未优先绑定折痕轴方向。
- 沿用旧的方向锁，未在姿态变化后清理，导致“上一姿态方向”污染当前姿态。
- “方向锁一次”被误用为“全生命周期只锁一次”，而不是“同一折痕轴周期只锁一次”。

### 通用修复方案
- 先根据折痕原始矩形 `width/height` 判定折痕轴，再决定方向组（固定方向法）。
- 方向策略改为“按折痕轴固定映射”：竖折痕使用翻转 90 度后的语义方向，横折痕保持原语义方向。
- 当 `foldStatus / foldDisplayMode / 折痕轴` 任一变化时，先清理旧方向锁，再重新决策方向。
- 页面退出时统一恢复 `AUTO_ROTATION`，避免跨页面串态。

## 场景 B-3：固定竖折痕设备被矩形假象误判为横折痕

### 问题描述
- 设备物理折痕为竖轴，但 `creaseRect` 在某些姿态下表现为横向宽条（例如 `2232x128`）。
- 页面本应“上展示下操作”，却被误分为左右分栏，悬停态语义错误。

### 根因分析
- 折痕轴仅依赖单次 `width/height` 比较，未引入设备固定折痕轴约束。
- 方向决策与分栏决策未做“方向到位再分栏”的时序保护，导致短时落入错误布局。
- 验证只看日志或截图单一证据，未同时核对布局树坐标，问题被误判为已修复。

### 通用修复方案
- 为目标设备补充显式约束（如 `fixedHingeAxis = vertical`），折痕轴判定按“语义/设备约束优先，几何兜底”执行。
- 命中竖折痕 + “上展示下操作”语义时，先切目标方向（旋转 90 度），方向未就绪时仅等待，不提前渲染分栏。
- 方向就绪后强制落上下分栏，再执行折痕分界重算，避免左右分栏回退。
- 验证至少覆盖悬停态与展开态两态，并同时保留 `snapshot_display + dumpLayout + 关键日志` 证据。

## 场景 C：折痕避让偏移或跨折痕

### 问题描述
- 分界线偏上/偏下，设备间表现不一致。
- 交互控件落入折痕区域，点击和阅读体验下降。

### 根因分析
- 使用经验比例切分布局，未读取真实折痕几何。
- 全局坐标与页面坐标混用，未完成坐标映射与 root 偏移扣减。
- 将 `16/40 vp` 直接当作分区边界锚点，导致分界线整体偏上或偏下。
- 旋转过程下 `convertGlobalToRelativeCoordinate` 结果出现轴互换/跳变，导致折痕厚度被误算为大面积区域。
- 缺少折痕厚度“异常值保险”，误把整屏或大面积内容当成折痕避让区。

### 通用修复方案
- 使用 `getCurrentFoldCreaseRegion()` 获取真实折痕。
- 使用坐标转换 API 时必须配套“旋转稳定性保护”；必要时回退到折痕原始矩形 + root 偏移的同单位换算路径。
- 先按折痕上/下（或左/右）边界落分区，再在分区内部应用 `16/40 vp` 安全间距。
- 分界线仅作为视觉辅助或中间量，不用“中线 + 固定偏移”直接推导容器边界。
- 增加折痕厚度保险阈值（例如不应超过可用高/宽的约 `35%`）；超阈值时回退到薄边估算，避免整屏被吞。

## 场景 C-2：悬停态黑屏（布局只剩 Blank）

### 问题描述
- 进入悬停后出现整屏黑色，内容区不可见。
- `dumpLayout` 中仍有页面根节点，但关键组件（视频/详情）缺失，仅保留大面积 `Blank`。

### 根因分析
- 折痕轴误判 + 折痕厚度误算叠加，导致“避让区”覆盖主要内容区。
- 方向锁和折痕重算链路不同步，出现“方向已切换、分区仍按旧轴计算”。

### 通用修复方案
- 将黑屏问题先归类为“折痕分区计算错误”，优先排查折痕轴判定与厚度计算链路。
- 在重算前清理失效方向锁，确保“方向决策”和“折痕分区”使用同一轴结论。
- 对折痕厚度做上下界约束并提供回退策略，禁止生成覆盖主内容区的异常折痕带。
- 验证必须同时采集截图与布局树：截图确认视觉结果，布局树确认关键组件是否仍在渲染树中。

## 场景 D：开合后连续性断档

### 问题描述
- 折展后出现步骤增加、滚动偏移、输入丢失、媒体进度异常。
- 暂停态在折展后被错误拉起播放，或播放态出现回退跳变。

### 根因分析
- 恢复链路只覆盖单一触发点（仅事件或仅 prepared）。
- 恢复标记提前失效，或 `autoplay` 抢跑覆盖业务状态。
- 恢复任务不可重入，导致多次重建时丢状态。

### 通用修复方案
- 使用“事件触发 + prepared 触发”的双触发恢复。
- 恢复完成前保持恢复标记有效，禁止固定短超时提前清空。
- pending restore 期间抑制 `autoplay` 抢跑；恢复后再交给业务状态。
- 恢复任务支持去抖、重入、取消，并在稳定后回收。

## 场景 E：生命周期回收缺失导致串态

### 问题描述
- 页面退出后仍响应旧监听，影响后续页面。
- 折展相关状态在跨页面后出现污染。

### 根因分析
- `display.on` / `window.on` 未与页面生命周期成对回收。
- 临时方向锁和恢复任务未在离场时清理。

### 通用修复方案
- 所有监听注册必须与对应 `off` 成对出现。
- 页面离场时统一清理定时器、恢复任务、方向锁和临时状态。
- 对跨页面共享状态增加作用域隔离，避免历史值串入新页面。

## 统一验证矩阵

- 状态维度：展开、半折叠、折叠三态切换。
- 路径维度：`y->z->y` 与 `y->p->y` 两条链路均验证。
- 连续性维度：滚动、输入、媒体进度与播放/暂停状态保持语义一致。
- 生命周期维度：页面进入/退出后无监听残留、无方向串态、无任务残留。
- 证据维度：`snapshot_display + dumpLayout + DisplayManagerService` 三份证据同时具备，禁止只看单一日志下结论。
