# 前置信息

用户提供了相关源码在code/目录。

用户提供了相关带符号表信息的so在lib.unstripped/目录。

用户环境提供llvm工具（llvm-readelf/llvm-addr2line/llvm-objdump等）。

**重要提示：** 需要分析调用栈和业务代码时，请先完成以下准备工作：

1. **行号解析**：需要解析获取代码行号信息，so文件放在lib.unstripped/目录
    - 在lib.unstripped/目录下查找对应so（不要输出查找过程）
    - 使用`llvm-addr2line`解出偏移对应的代码行号
    ```sh
    llvm-addr2line -Cfpie path/to/so offset
    ```
    - `ld-musl-xxx.so`即`libc.so`

2. **代码定位**：确认代码放在code/目录

按以下步骤分析问题：

1. 基本信息分析
2. 任务队列分析
3. 阻塞链跟踪
4. 流水日志分析
5. 综合分析与报告

# 基本信息分析

1. **确认故障类型**
    - 检查`Reason`字段，确认是`SERVICE_BLOCK`还是`THREAD_BLOCK`
    - 理解对应问题的检测原理和阈值

2. **记录关键信息**
    - `TIMESTAMP`：故障发生时刻，用于在hilog日志中锁定时间范围
    - `TID`：发生问题的线程ID，问题分析从此线程开始

3. **查看整机资源状态**
    - `ReclaimAvailBuffer`：整机剩余可用内存，判断是否低内存问题（注意内存信息抓取的时间，如果在freeze时间后不具参考价值）
    - `ThermalLevel info`：热等级大于等于5时，系统可能受热限频影响性能
    - 如果存在整机资源问题且业务分析无异常，可优先考虑问题与之相关

# 任务队列分析

1. **分析当前任务**
    - 查看两次检测（SERVICE_WARNING/SERVICE_BLOCK，或THREAD_BLOCK_3S/THREAD_BLOCK_6S）的`Current Running`任务
    - **判断规则**：
      - 如果两次检测任务相同：说明是同一任务阻塞
      - 如果两次检测任务不同：需要根据任务队列信息分析耗时较长的任务

2. **验证时间合理性**
    - 假设最后一个watchdog任务在T时刻提交
    - 第一次抓栈：SERVICE_WARNING预期时间T+5s，THREAD_BLOCK_3S预期时间T+3s
    - 第二次抓栈：SERVICE_WARNING预期时间T+10s，THREAD_BLOCK_6S预期时间T+6s
    - **异常判断**：如果实际时间与预期时间差异较大，说明DFX抓取有时间偏差，可能存在低内存/高负载等整机问题

3. **分析任务队列**
    - 检查各优先级任务队列（VIP、Immediate、High、Low、Idle）
    - 重点关注：
      - 任务提交时间（send time）
      - 任务执行时间（trigger time 开始，comleteTime 完成）
      - 任务优先级（priority）
      - 任务提交者（caller）
      - 任务名称（task name）

# 阻塞链跟踪

根据两次抓取的调用栈分析线程阻塞关系，分析时必须解栈。

## 调用栈对比

1. **对比两次抓栈**
    - 两次抓栈一致：卡在同一业务
    - 两次抓栈不一致：可能抓到不同任务的两次瞬时栈，或者未卡住但执行时间长的同一任务

2. **分析线程状态**（如果有）
    - 查看线程状态字段：`state`，`utime`，`stime`，`priority`，`nice`，`clk`
    - **关键判断**：
      - 如果线程栈顶未等待资源/陷入内核但未处于执行状态（state=S），可能是整机问题（低内存/IO问题等）导致
      - 结合`utime`/`stime`和抓栈时间间隔，确定线程在用户态和内核态的执行时长，分析线程负载情况
      - 对比两次抓栈的运行时间，如果无变化则说明线程未被调度

3. **分析堆栈阻塞模式**
    - 识别栈顶的阻塞函数，参考[常见阻塞模式](references/blocking-patterns.md)

## 阻塞链分析

**重要提示：** 分析从基本信息中的`Tid:<TID>`线程开始。

1. **识别阻塞类型**
    - 根据栈顶函数判断阻塞原因（等锁、信号量、IPC调用等）

2. **查找阻塞源头**
    - 栈顶等锁/信号量/条件变量：需要找到当前等待资源的持有线程或触发线程
    - 栈顶为IPC调用：需要找到ipc对端线程
    - 阻塞线程通常与被阻塞栈顶在同一so或者同一个类的业务中，找到后分析对应线程的堆栈信息

3. **递归分析阻塞**
    - 如果阻塞线程也处于阻塞条件，继续分析阻塞其的线程
    - 直到找到不处于以上阻塞状态的线程，作为最终问题线程，根据调用栈分析其业务

如果是ipc阻塞，检查对端信息分析阻塞情况：
- 查看`BinderCatcher`信息，获取源为`当前进程号:当前线程号`的同步调用，查看阻塞时间
- 查看`对端进程号:对端线程号`信息，查找`PeerBinder Stacktrace`中对应的进程抓栈
- 从对端线程号对应的线程开始分析阻塞情况

# 流水日志分析（如果有日志）

1. **获取流水日志**
    - 检查是否存在`hilog`和`hilog_kmsg`文件

2. **分析流水日志**
    - 分析在`TIMESTAMP`问题发生时间附近的日志
    - 识别可能相关的异常日志，参考 [常见流水日志](references/hilog-refernce.md)
    - 识别业务场景和操作流程，参考 [常见流水日志](references/hilog-refernce.md)

# 综合分析与报告

1. **确定根本原因**
    - 综合堆栈信息、任务队列、线程状态、资源状态
    - 判断问题的相关业务信息，或者相关的整机问题

2. **绘制阻塞路径**
    - 描述从阻塞线程开始到最终问题线程的完整调用链
    - 标注关键阻塞点和等待的资源

3. **分析业务信息**
    - 解析堆栈对应的业务代码
    - 结合关键日志，说明卡住的业务场景和操作流程

4. **生成分析报告**
    - 故障分析总结
    - 阻塞路径分析
    - 问题根本原因（如果结果不完整，则给出下一步排查方向）
    - 建议优化方向

**重要提示**：完成分析生成报告并保存到文档。
