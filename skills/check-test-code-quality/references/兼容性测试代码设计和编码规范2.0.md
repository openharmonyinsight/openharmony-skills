1 代码目录管理

**1.1 测试套分类定义** 
|  套件分类 |  英文描述 | 类型编码  |  测试套件类型定义说明 |
|---|---|---|---|
|  OpenHarmony应用兼容性测试套 | Application Compatibility Test Suite  |  acts | 验证OEM厂商实现的OpenHarmony平台符合Openharmony的API接口定义，满足兼容性要求，看护要点：SDK API、NDK API、关键架构；  |
| OpenHarmony分布式兼容性测试套  | Distributed Experience Test Suite  | dcts  |  验证OEM厂商实现的OpenHarmony平台没有修改Openharmony分布式框架与定义，确保分布式兼容； |
| OpenHarmony硬件抽象层兼容性测试套  |  Hardware Abstraction Test Suite | hats  |  验证OEM厂商实现的OpenHarmony平台符合 Openharmony的HDF架构，HDI API定义；|

 **1.2 代码目录** 
 
 **规则1.2.1** ：兼容性测试代码必须按照对应的目录结构要求归档
【正例】 目录规划样例：
/test/xts
├── acts                  #  测试代码存放目录
│   └── subsystem         #  标准系统子系统测试用例源码存放目录
│   └── subsystem_lite    #  轻量系统、小型系统子系统测试用例源码存放目录
│   └── BUILD.sh          #  标准系统测试代码编译入口
│   └── test_packages.gni #  标准系统测试代码编译配置
│   └── build_lite        #  轻量系统、小型系统测试用例编译配置存放目录
│       └── BUILD.gn      #  轻量系统、小型系统测试用例编译配置


**规则1.2.2** ：测试代码目录[subsystem_name]要求与OpenHarmony的子系统英文名称保持一致。

**建议1.2.1** ：测试代码目录命名全部采用小写字母，文件名称命名最多不超过30英文字符。

 **2 测试代码编程规范** 

2.1 编程语言约束

2.1.1 目的
兼容性测试代码作为OpenHarmony重要看护手段，需要具备开发代码所需要具备的可读、可维护、安全、可靠、可测试、高效、可移植七个特征，还应该具有一致的代码风格、易读、高质量的Clean Code。

2.1.2 通用语言约束

**规则2.1.2.1** ：兼容性测试代码必须满足OpenHarmony编程规范要求。
ArkTS语言编程规范：https://gitcode.com/openharmony/docs/blob/master/zh-cn/contribute/OpenHarmony-Application-Typescript-JavaScript-coding-guide.md


2.3 测试代码设计规范

**规则2.3.1** ：测试代码设计要意图明确、逻辑简单、检查点单一。

**规则2.3.2** ：测试代码只能基于操作系统已公开Public API、HDI API、Test API进行测试。 

**规则2.3.3** ：以函数API定义/文档为依据设计测试用例，不能以函数API实现为依据进行测试设计。

**规则2.3.4** ：函数API声明的功能描述、定义的入参、出参、 返回值、权限均需设计检查点。

**规则2.3.5** ：函数API 声明有可选参数，测试代码必须验证可选参数默认值即不输入可选参数的场景。 

**建议2.3.1** ：测试代码针对函数API的入参，建议参考下表针对不能数据类型的入参进行测试设计。
| 入参类型  |  测试设计建议 |
|---|---|
|  数值类型 | 如限定取值范围，需使用等价类、边界值方法针对取值范围进行测试设计（边界值、合法值、非法值），且需验证返回值内容是否符合预期。  |
|   |  针对入参异常值如负数等进行验证，验证返回错误码是否符合预期。 |
|   |  针对数据类型最大数值等进行验证，针对数值翻转场景测试覆盖 |
| 字符串类型  | 如限定输入内容，需根据等价类方法进行测试设计，且需验证返回值内容是否符合预期。  |
|   |  如限定输入长度，需根据边界值方法进行测试设计，需验证返回值内容是否符合预期。 |
|   |  如未限定输入长度，需验证超长字符串场景下接口功能，并作为规格基线长期看护； |
|   |  如限定输入字符类型如字母、数字等，需根据等价类方法进行测试设计； |
|   |  如是数据库、URL相关函数，需设计转义字符串进行验证 |
|   |  需有异常输入测试设计，如null、“”等 |
| 结构体类型  |  需构造不同数据下的对象及空对象输入进行测试验证 |
| 布尔类型|  需遍历输入布尔型取值测试验证 |
| 枚举类型  |  必须覆盖全部取值 |


**建议2.3.2** ：测试代码针对函数API出参、返回值，参考下表进行设计。
| 场景分类  |  测试设计参考 |
|---|---|
| 如API存在出参  | 需验证API的不同出参对应的功能逻辑，并需验证出参的具体内容是否符合预期；  |
| 如API存在返回值  |  需验证API的不同返回值对应的功能逻辑，并需验证返回值的具体内容是否符合预期； |
| 如API不存在出参或返回值  | 需通过接口功能对周边模块影响进行验证  |
| 如API功能对其他模块或其他API逻辑产生影响  | 需要验证该影响，不可以仅验证出参、返回值  |


**规则2.3.6** ：若函数API同时支持 callback 方式和 Promise 方式，需要分别设计测试用例。
【正例】 如下为callbak方式与Promise方式，需要分别设计测试测试用例：
function 方法名(callback: AsyncCallback<结果数据类型>): void;
function 方法名(): Promise<结果数据类型>;

**规则2.3.7** ：若函数API存在抛出异常场景，需要使用try/catch设计测试用例验证异常，包括错误码、错误信息需要设计检查点。

**规则2.3.8** ：若订阅事件API存在事件名枚举，需要分别设计测试用例验证。
【正例】
function on(type: 事件名枚举, callabck: 回调函数类型): void;

**规则2.3.9** ：枚举API需要使用对象引用方式分别设计测试用例验证。

**规则2.3.10** ：若属性API存在默认值，需要设计测试用例验证。

**规则2.3.11** ：如API是组件API，必须针对组件支持的操作类型进行覆盖测试。

**建议2.3.3** ：如API是组件API，建议覆盖该组件和其他组件的组合嵌套场景。

**规则2.3.12** ：若函数API存在权限控制，需针对不申请权限调用场景进行验证，确保存在权限管控。

**规则2.3.13**：若函数API存在并发安全标记（concurrentsafe），需设计并发场景测试用例验证。

**规则2.3.14**：测试用例的测试对象是操作系统，不允许把某个具体产品的具体状态作为检查点，但可针对通用品类的不同实现进行分支逻辑处理。

**规则2.3.15**：使用SystemCapability能力进行部件差异化的测试用例设计，使用canIUse接口设备是否支持某个特定SysCap能力的判断；

**规则2.3.16**：针对品类产品特有部件能力，要求独立设计测试套工程测覆盖。

**规则2.3.17**：测试用例间必须解耦，不能存在执行依赖，公共依赖部分按照2.6章节描述抽离。

2.4 测试代码注释规范
【说明】测试代码中用例编号、用例粒度、测试类型、用例级别、开源声明等用例基本属性通过测试代码注释方式管理。

**规则2.4.1**：新增测试代码文件必须有开源说明信息。

**规则2.4.2**：用例编号的注释“@tc.number”字段必填，且用例编号保持唯一。

**规则2.4.3**：用例编号命名规则`sub_{子系统}_{部件}_[XX?]_增加4位阿拉伯数字标识，用例编号的递增要求以100为单位

【正例】测试用例编号样例：SUB_APPEXECFWK_BUNDLEMGR_SDK_HAG_0100

**规则2.4.4**：用例名称的注释“@tc.name”字段必填，且与测试方法名保持一致。

**规则2.4.5**：用例描述的注释为“@tc.desc”字段必填，描述用例的测试场景等信息。

**规则2.4.6**：用例级别字段要求在函数定义中必填。

【正例】

it("testAttributeIt", Level.LEVEL0, () => {
     console.info('Hello Test');
   })

**规则2.4.7**：用例规模字段要求在函数定义中选填，默认为中型测试。
【正例】
it("testAttributeIt",  Size.SMALLTEST, () => {
     console.info('Hello Test');
   })

**规则2.4.8**：测试类型在函数定义中选填，默认为功能测试
【正例】
it("testAttributeIt",  TestType.FUNCTION, () => {
     console.info('Hello Test');
   })

2.5 测试代码实现规范

2.5.1 测试工程目录命名

**规则2.5.1.1**：测试工程目录/文件夹统一采用小写英文风格命名，不允许出现中文。

2.5.2 测试源码文件命名

**规则2.5.2.1**：C++测试源码文件名称以被测对象英文描述+Test后缀命名，统一采用大驼峰风格。
【正例】C++测试源码文件命名样例：IntentFilterTest.cpp

**规则2.5.2.2**：C测试源码文件名称以被测对象英文描述+_test后缀命名，英文单词间用下划线分割。
【正例】C测试源码文件命名样例：intent_filter_test.c

**规则2.5.2.3**：JavaScript测试源码文件名称以被测对象英文描述.test后缀命名，统一采用大驼峰风格。
【正例】JavaScript测试源码文件命名样例：Intent.test.js

**建议2.5.2.1**：测试源码文件名称在相同测试工程中唯一，避免重复。

2.5.3 测试用例命名

**规则2.5.3.1**：测试用例命名格式： test[测试点]，测试点采用驼峰描述，测试用例首字母小写。

2.5.4 预置条件

**规则2.5.4.1**：依赖测试环境/测试资源的测试套件，在SetUp中增加检查点，检查测试环境/测试资源准备是否满足要求。

2.5.5 用例步骤

**规则2.5.5.1**：相同测试套件中测试用例间不能相互依赖。
【说明】不同的测试用例之间不能有相互依赖关系，如测试用例1依赖测试用例2的执行结果。

**建议2.5.5.1**：测试代码设计要求实现逻辑与数据分离。

2.5.6 期望结果

**规则2.5.6.1**：测试用例必须使用测试框架提供的断言描述测试结果。

**规则2.5.6.2**：测试用例期望结果检查必须准确全面。

**规则2.5.6.3**：测试用例不允许没有检查点，或有恒真检查点。

2.5.7 环境清理

**规则2.5.7.1**：测试代码执行完毕必须进行环境清理，避免对其他测试用例造成环境污染。清理内容包括配置项、测试资源、预置条件恢复等。

2.5.8 其他

**规则2.5.8.1**：测试代码中不允许出现无用注释代码。

**规则2.5.8.2**：测试代码中不允许出现无用导包。

**规则2.5.8.3**：测试代码中不允许出现拼写错误。

2.6 测试基础库设计规范

**规则2.6.1**：多个测试套件共用的测试方法可封装为可复用的测试基础库。

**规则2.6.2**：测试基础库按子系统维度管理，在子系统测试工程根目录新建utils目录。

2.7 测试代码修改规范

**规则2.7.1**：已发布兼容性测试代码的修改必须有框架TMG或相关技术组织的变更评审结论，否则不允许修改。

3 测试代码配置规范

3.1 编译脚本配置

**规则3.1.1**：测试工程需提供按操作系统类型、芯片平台类型编译测试套件的编译脚本。

3.2 测试代码配置

**规则3.2.1**：测试模块二进制命名格式要求：`TestSuiteTypeTest`; [TestSuiteType]；分别为“acts”, “dcts”, “hats”，[ModuleName]为被测模块名称，采用大驼峰风格描述。

【正例】acts测试模块命名：ActsUtilsTest

**规则3.2.2**：每个测试模块均需定义执行依赖的Test Runner、测试资源以及测试模块加载安装配置等基础运行依赖的测试配置参数。

**规则3.2.3**：新增测试工程中，必须在工程的BUILD.gn文件中配置”part_name “以及”subsystem_name”字段，配置值参考OpenHarmony系统定义子系统和部件英文名称。

**规则3.2.4**：测试工程配置文件Test.json中，不允许配置影响系统状态的预处理命令，如关闭系统SELinux、关闭日志检查等。

4 第三方开源测试代码引入规范

**规则4.1**：测试代码若要引入新的第三方开源软件到OpenHarmony项目中，请参考第三方开源软件引入指导。

**规则4.2**：第三方开源软件必须适配兼容性测试套的编译命令。

**规则4.3**：第三方开源软件需要对接xdevice执行框架，可以统一测试执行、结果呈现，log输出。

5 测试代码维护管理

**规则5.1**：测试代码仓的演进和发布，跟随主干源码演进和发布。

**规则5.2**：测试代码上库前必须通过代码门禁检查。

**规则5.3**：测试代码上库PR 必须附录本地测试执行报告。
